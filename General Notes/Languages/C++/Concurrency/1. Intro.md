# Thread vs Process
- Process
	- Instance of a running program
	- Has own
		- Virtual address space
		- File descriptor table
		- Signal handlers
		- PID
	- PID manages separate page table for each process
		- Process cannot read/write to another's memory
	- Context switching is more expensive
		- Flush TLB as page tables change
			- TLB miss cascade after process switch can cost thousands of cycles while cache lines get re-fetched from page table
	- Communication is harder
		- Explicit IPC - pipes, Unix domain sockets, shared memory segments (mmap, shmget), message queues, signals
			- Involves kernel transitions (syscalls) or careful syncing
- Thread
	- Execution context within a process
	- Has own
		- Stack
		- Registers
		- Program counter
		- Thread-local storage (`thread_local`)
	- Shares
		- Virtual address space
		- File descriptors
	- Context switching is cheaper
		- Just swap out register file, kernel stack pointer and thread-local storage base
	- Communication is easier
		- threads share memory by default

# Why concurrency
- Separation of concerns
	- Each thread does one thing
- Performance
	- Task parallelism
		- Split task up and do at the same time
	- Data paralellism
		- Split data up and handle at the same time
- To avoid blocking program progress due to slow I/O
	- eg. Writing program that does different kinds of I/O
	- We want to do something else while waiting for I/O. CPU scheduler can switch to other threads which are ready to run and do something useful
	- **Threading** enables **overlap** of I/O with other activities within a **single program**

# Why not concurency
- If perfomance gain not worth
	- Thread has overhead too, as OS has to allocate kernel resources, stack space, add thread to scheduler tetc
	- Threads are limited resources
		- Have their own stack. (1MB stack -> if 4GB memory, used up by 4000+ stacks)
