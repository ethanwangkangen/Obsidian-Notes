# `std::future`
- Single use/one shot
- Read end of a one-time asynchronous result channel
- Consumer's handle to a value/exception that will be produced elsewhere
	- By `promise`, `packaged_task`, `async`
- Shares heap-allocated **shared_state** with whichever producer created it
- How to get
	- promise.get_future() — manual producer
	- packaged_task.get_future() — wrapped callable
	- std::async(...) — returns a future directly
- Retrieving the result
	- `.get()`
	- Blocks until result its ready
	- **Single use** and **move only**
- Checking state without consuming
	- `valid()` - true while future has shared state, becomes false after `get()`
	- `wait()` - block until ready but **don't retrieve** (can still `get()` after) 
- Async-future-blocking destructor
	- Behaviour upon falling out of scope without .get()/.wait() called differs based on how it was obtained
		- std::async - destuctor **blocks** until the std::async task finishes
		- std::promise - destructor **does not block**, just releases resources
# `std::async()`
- Used to start **asynchronous** task for which don't need result right away
- Returns a `std::future` object which will eventually hold the **return value** of the function
- `.get()` to get value
	- Thread blocks until future is *ready*, and returns the value
```cpp
#include <future>
std::future<int> answer = std::async(someFunction);
//...
cout << answer.get();
```
- Same rules for passing in arguments as `std::thread`, see [[2. Thread Management]]
- Behvaiour
	- By default, up to implementation if it starts a new thread of runs synchronously when future is waiter for
	- Can specify additional parameters to specify
		- `std::launch::async`
			- Must run on own thread
		- `std::launch::deferred`
			- Defer until `wait()` or `get()` is called
		- `std::launch::async | std::launch::deferred`
			- Implementation may choose

# Exceptions
- Exceptions in threads must be properly propagated to the waiting thread
- `std::future` allows exceptions thrown in asynchronous tasks to be captured
	- Then **rethrown** when future's `get()` method is called

# `std::promise()`
- Also returns a `std::future<>` upon `.get()`, but is the **write** end 
	- Producer holds a promise and eventually puts something in
	- Consumer holds `std::future<>` and blocks until something arrives
	- They share a heap-allocated shared state that holds the value/exception plus the synchronisation (flag and condition-variable-like-wait)
```cpp
std::promise<int> prom;                  // creates the shared state
std::future<int> fut = prom.get_future(); // hands you the read end (ONCE)

// producer thread:
prom.set_value(42);                      // writes value, marks ready, wakes waiters

// consumer thread:
int x = fut.get();                       // blocks until ready, then returns 42
```
- Resolves promises in 4 ways
	- `set_value(v)` 
		- Fulfils it
		- The future's `get()` unblocks and returns `v`
	- `set_exception(eptr)`
		- Stores an exception instead
		- Future's `get()` rethrows it in consumer thread
	- Broken promise
		- If promise is destroyed without ever setting a value or exception, the shared state is marked with `std::future_error` of code `broken_promise`, and the waiting `get()` throws it
		- Consumer never hangs forever just because producer died/forgot
	- Single-use enforcement
		- Setting the value/exception twice throws `promise_already_satisfied`



# `std::shared_future`
- Result can be read multiple times and by multiple threads
- **Copyable** instead of just moveable 
	- (in contrast to `std::future`, only one instance refers to a particular async result at a time)
- `get()` is **non consuming**
- How to make
	- `fut.share()` -> moves a plain future's state into a shared_future
		- invalidates the original future
		- `std::shared_future<T> sf = promise.get_future().share();`
		- or `std::shared_future<T> sf = std::move(fut)`
- Thread safety
	- The **shared state** is thread safe
		- The heap-allocated block that actually holds the result value, the exception, ready flag and synchronisation machinery
		- Exactly **one** of these created by the promise, referred to by every future/shared_future tied to it
	- The **handle** is **not** thread safe
		- The `std::shared_future` object itself
		- ie. cannot have multiple threads using the same `shared_future` object, will result in data race
		- Solution
			- Each thread has its own `shared_future`
		- Similar to **shared_ptr**
			- The control block is thread safe but a single shared_ptr instance is not safe to mutate from multiple threads
```cpp
// DANGEROUS: one handle, many threads
std::shared_future<int> sf = p.get_future().share();
for (int i = 0; i < 4; ++i)
    threads.emplace_back([&sf]{ sf.get(); });   // all touch the SAME sf object
    
// SAFE: one handle per thread, all referring to the same shared state
for (int i = 0; i < 4; ++i) {
    std::shared_future<int> my = sf;            // copy → separate handle object
    threads.emplace_back([my]{ my.get(); });    // each touches its OWN handle
}
```
