# Atomic operations
- Non-atomic `x++` is actually load -> modify -> store (3 steps), and another thread can interleve in between them
	- Data race
- Atomic -> indivisible
	- No thread can ever see it half complete
- 3 distinct guarantees
	- Indivisibility of read-modify-write
	- No tearing of a single load/store
	- Ordering and visibility - controlled by operation's `memory_order`
		- This is what lets one thread's writes be visible to another in a defined way

# Atomic types
## Lock free?
- Most are lock free
	- Can check with `is_lock_free()` member function (runtime)
- Why relevant?
	- If not free, then defeats the point of replacing mutex-protected operations with something faster
- Compile time lock free info (C++17)
	- `std::atomic<T>::is_always_lock_free` **static constexpr member variable** of all atomic types
		- true if this type is lock-free on **every run** on **all supported** platforms (so can `static_assert` or branch at compile time)
		- because eg. `std::atomic<uintmax_t>` might only be lock free if hardware used supports necessary instructions, so this is a runtime property, and `is_always_lock_free` returns false

## The only guaranteed type - `std::atomic_flag`
- Operations on this type are **always** lock-free
	- Is the ONLY type that is guaranteed lock free
- All other atomics can be built on top of his
- Has no `is_lock_free()` function

## Naming/type set
- Generic template is `std::atomic<T>`
	- `std::atomic<bool>`
	- `std::atomic<char>`
	- `std::atomic<unsigned long>`
	- etc.
- Standard provides named typedefs
	- `std::atomic_bool`, `std::atomic_int` etc.

## Non-copyable, non-assignable (between atomics)
- **No copy constructor** and **no copy assignment** from another atomic
	- Because copying would require 2 atomic operations (load one, store other)
- Can assign from the **underlying value type**
	- `atomic<bool> b;` `b = true;`
		- Maps to a `store`
		- The assignment returns the **plain value**, **NOT a reference to the atomic!!**!
- Likewise reading via implicit conversion (`bool v = b;`) maps to a `load`
- All real work is done through **member functions** `load`, `store`, `exchange`, `compare_exchange_weak/strong`, and `fetch_*`

## Macros
- Types
	- `ATOMIC_BOOL_LOCK_FREE`,
	- `ATOMIC_CHAR_LOCK_FREE`,
	- `ATOMIC_CHAR16_T_LOCK_FREE`,
	- `ATOMIC_CHAR32_T_LOCK_FREE`,
	- `ATOMIC_WCHAR_T_LOCK_FREE`
	- `ATOMIC_SHORT_T_LOCK_FREE`
	- `ATOMIC_INT_LOCK_FREE`
	- `ATOMIC_LONG_LOCK_FREE`
	- `ATOMIC _LLONG_LOCK_FREE`
	- `ATOMIC_POINTER_LOCK_FREE`
- Specify the lock-free status of the corresponding atomic types for the specified ubilt-in types and their unsigned types
- Evalue to 
	- 0 if never lock free
	- 1 if lock-free is runtime property
	- 2 if always lock free
## Memory-order parameter
- Every atomic **member function** optionally takes a `std::memory_order`
	- Default is `std::memory_order_seq_cst`
- 6 orders
	- `memory_order_relaxed`
	- `memory_order_consume`
	- `memory_order_acquire
	- `memory_order_release`
	- `memory_order_acq_rel`
	- `memory_order_seq_cst`
- Which are legal depends on the operation kind.
	- **3 types:**
		- **Stores** (`store`) - WRITE
			- `relaxed`, `release`, `seq_cst`
		- **Loads** (`load`) - READ
			- `relaxed`, `consume`, `acquire`, `seq_cst`
		- **Read-modify-write** (`exchange`, `fetch_*`, `compare_exchange_*`) 
			- Any 

## Operations on std::atomic_flag
- Simplest type, basic building block only
- **Must** be initialised with ATOMIC_FLAG_INIT
	- Regardless of scope and where it's declared
	- Initialises the flag to a **clear** state
	- `std::atomic_flag f = ATOMIC_FLAG_INIT;`
- 2 operations
	- `clear()` - is a **store** operation 
		- atomically clears flag to **false**
	- `test_and_set()` - is a **read-modify-write** operation
		-  sets flag to **true** and returns its **PREVIOUS** value (a bool)
- NO load operation
	- Cannot read without writing
- Again, because atomic, cannot copy assign/construct
```cpp
f.clear(std::memory_order_release);
bool x = f.test_and_set();
```

### Spinlock mutex with std::atomic_flag
```cpp
class spinlock_mutex
{
public:
	spinlock_mutex() :
		flag(ATOMIC_FLAG_INIT)
	{}
	void lock()
	{
		// test_and_set returns the OLD value.
		// so this spins until the previous value is false, ie unlocked.
		// then we set it to true
		while (flag.test_and_set(std::memory_order_acquire));
	}
	
	void unlock()
	{
		// set to false
		flag.clear(std::memory_order_release);
	}

private:
	std::atomic_flag flag;
}
```
- True == locked, false == unlocked
- Spin while locking
	- test_and_set() -> checks previous value. Once previous value becomes false (unlocked), take control and lock (set to true)
- Unlock
	- clear() -> set to false
- This is a busy-wait lock

# Operations on `std::atomic<bool>`
- More useful boolean than `atomic_flag` - has a real load and store
- Still not copy-assignable from another atomic, but assignable from a plain `bool`
```cpp
std::atomic<bool> b (true); // construction is not atomic
b = false; // sugar for b.store(false); (atomic)

bool result = (b = false);// NOTE: assignment operator returns a bool instead of a reference to the object!
```


- Operations (for the legal members, check the rules for store/load/rmw above)
	- `store(bool, order = seq_cst)` - **store**
	- `load(order = seq_cst)` - **load**
	- `exchange(bool, order = seq_cst)` - **rmw**
	- `compare_exchange_weak / compare_exchange_strong` — the workhorse (below).

### Compare-exchange (CAS) - the cornerstone of lock-free programming
```cpp
// One order CAS form:
// member function of std::atomic<bool>
bool compare_exchange_strong(T& expected, T desired,
                             std::memory_order order = seq_cst);
```
**IMPORTANT**
- If the atomic's **current value** == `expected`
	- Store `desired` into the atomic
	- Return `true`
- Else
	- Load atomic's **actual current value** into `expected`
	- Modifies the `expected` variable! 
	- Return `false`

### Weak vs strong
- `compare_exchange_strong`
	- returns `false` **only** when the value genuinely didn't match `expected`. No false negatives.
- `compare_exchange_weak`
	- May **fail spuriously**
		- Return `false` _even when the value did equal `expected`_.
		- Happens on architectures using **load-linked/store-conditional (LL/SC)** (ARM, PowerPC)
			- store-conditional can fail if the cache line was touched (even by an interrupt or context switch) between the load and store.

> Use **`_weak` inside a loop** (you're already retrying, so a spurious failure just costs one extra iteration — and `_weak` compiles to a tighter sequence on LL/SC machines, no internal loop).

> Use **`_strong` when you do NOT have a natural loop** (a single decisive attempt), because `_strong` itself loops internally to mask spurious failures — so writing your own loop around `_strong` doubles the looping.

### Looping
- As mentioned above, weak exchange must be used in a loop

```cpp
bool expected = false;
exten atomic<bool> b; // set somewhere else
while (!b.compare_exchange_weak(expected, true) and !expected);
```
- What is the loop trying to do
	- Keep trying to flip `b` from false to true until either I succeed or someone else beats me tos etting it true
- Why 2 conditonals here?
	- !b.compare_exchange_weak(expected, true) — "the CAS failed."
	- !expected — "expected is still false."
	- We want the loop to continue **only when both** are true, ie CAS failed AND expected is still false
	- Remember that compare_exchange_weak can fail (return false) for **2 different reasons**
		- b's actual value was **not** equal to `expected`
			- This means some other thread set b to true while we were looping. 
			- On this kind of failure, CAS writes b's **actual value into `expected` - so expected becomes true**
		- OR **spurious failure**. b actually did equal expected (both false). 
			- The CAS should have succeeded, but failued due to LL/SC architecture quirk
			- b is unchanged, `expected` is unchanged

In contrast, the strong version doesnt need a loop
```cpp
bool expected = false;
bool i_set_it = b.compare_exchange_strong(expected, true);
// b is now definitely true.
// i_set_it == true  -> I changed it from false to true
// i_set_it == false -> it was already true (expected now holds true)
```

| Outcome          | CAS returns | `expected` after    | what we want                            |
| ---------------- | ----------- | ------------------- | --------------------------------------- |
| Success          | `true`      | unchanged (`false`) | **stop** (I did it)                     |
| Spurious failure | `false`     | unchanged (`false`) | **retry** (nothing happened, try again) |
| Real failure     | `false`     | updated (`true`)    | **stop** (someone else did it)          |
- **The second column is only applicable for weak exchange.**
- CAS just returns whether or not the thread itself was responsible for successfully changing the value to desired.

2 Order CAS Form
- Can supply memory order appliyed if CAS succeeds vs fails
```cpp
bool compare_exchange_strong(T& expected, T desired,
                             std::memory_order success,
                             std::memory_order failure);
```
- **`success`** — memory order applied if the CAS succeeds (the store happens).
- **`failure`** — memory order applied if it fails (only a load happens).
- Constraints on `failure`:
	- it must **not** be `release` or `acq_rel` (a failed CAS performs no store, so it can't release)
	- it must **not be stronger than `success`**

## std::atomic<T*>
- Pointer arithmetic
- Operations
	- Same as `atomic<bool>`
	- `load()`
	- `store()`
	- `exchange()`
	- `compare_exchange_weak()`
	- `compare_exchange_strong()`
	- **`fetch_add()`** - RMW
		- Atomically advance pointer by `n` elements (compiler scales by sizeof(T)), return the **previous pointer value**
	- **`fetch_sub`** - RMW
		- Same as above but retreating
	- Operator forms `+=`, `-=` etc.
		- But these return the **new** pointer value not the old.

```cpp
Foo arr[5];
std::atomic<Foo*> p (arr);
Foo* x = p.fetch_add(2); // returns old pointer
assert (x == arr); 
assert (p.load() == &arr[2]);
```
## standard atomic integral types
- `atomic<int>`, `atomic<unsigned>`, `atomic<size_t>` etc..
- Operations
	- Everything from `atomic<bool>` plus:
	- `fetch_add(n)` / `fetch_sub(n)`
		- add/subtract, return **previous** value. 
	- `fetch_and(n)` / `fetch_or(n)` / `fetch_xor(n)` 
		- bitwise ops, return **previous** value
	- Compound assignment operators `+=`, `-=`, `&=`, `|=`, `^=` 
		- perform the op and return the **NEW** value.
	- `++` / `--` (pre and post) 
		- increment/decrement; pre returns new, post returns old, as usual. 

>`fetch_*` return the **OLD** value; compound operators and `++/--` (pre) return the **NEW** value. In lock-free code you almost always want `fetch_*` because you need to know _what the value was_ before your contribution.


## User-defined atomic types
- Using `std::atomic<>` primary class template
-  Operations
	- `load`
	- `store`
	- `exchange`
	- `compare_exchange_weak/strong`
	- NO arithmetic, bitwise ops..

- Can instantiate `std::atomic<UDT>` for your own type UDT
	- But only **if** UDT is *trivially copyable*
		- **No** user-provided **copy/move** constructors or assignment
		- **No** user-provided **destructor**
		- **No** **virtual** functions or virtual bases
		- Every base and non-static member is itself **trivially** copyable
- Why this restriction
	- Atomic operations move the object's bytes around with a `memcpy` - equivalent and compare them with a `memcmp`-equivalent
		- Non-trivial copy ctor or destructor bypassed so standard forbids
	- Since `atomic<UDT>` usually can't be lock-free, the library protects it with an `internal lock`
		- Calling user-defined copy/comparison operator would mean passing protected data into user code while holding the lock
	- Deadlock/blocking risk
		- Library may use **one shared lock** for all locking atomic operations
	- Enables lock free codegen
		- Terating the type as **raw bytes** increases the chance the compiler can use real atomic instructions


- **CAS uses bitwise `memcmp`, not `operator==`.*
	- Two values that are _logically equal_ but differ in **padding bytes** (uninitialized struct padding) or representation will compare **unequal**, causing CAS to fail even though "the value is the same." 
	- eg. (e.g. `+0.0` vs `-0.0`, NaN encodings)
> Avoid types with padding in CAS-heavy code, or zero-initialize padding.

- **Floating point**: `atomic<float>`/`atomic<double>` 
	- CAS compares the bit pattern, so `+0.0` and `-0.0` compare unequal, and a NaN never equals itself bitwise the way `==` would behave — CAS on floats can behave surprisingly. 
	- (Arithmetic `fetch_add` on floats was added in C++20.)

- If UDT is large or the platform can't do it atomically in hardware, `std::atomic<UDT>` falls back to an **internal lock** (`is_lock_free()` → false), and you lose the performance benefit. 

### What you can and can't wrap
- **Can't:** `std::atomic<std::vector<int>>` — `vector` has a non-trivial copy constructor/assignment, so it's not trivially copyable.
- **Can:** classes holding **counters, flags, pointers, or arrays of simple data** — trivially-copyable aggregates.
- **Rule of thumb:** the more complex the structure, the more you'll want operations beyond assignment/compare anyway — at which point use a **`std::mutex`** to protect it (chapter 3), not an atomic.
