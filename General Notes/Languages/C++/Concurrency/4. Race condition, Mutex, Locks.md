# Definition
- If **outcome** depends on **relative ordering of execution** of operations on 2 or more threads

# Avoiding
- Wrap data structure with protection mechanism
	- Ensure that only the thread performing a modification can see the intermediate states where invariants are broken
	- From POV of other threads, these modifications haven't started/have completed
- Design data structure and invariants so modifications are done as indivisible changes, each of which preseres invariants
	- Lock-free programming

# Mutexes
- Mark **piece of code** that access data structure as **mutually exclusive** so if any thread is running one of them, others must wait
```cpp
#include <mutex>
std::mutex myMutex;
std::list<int> myList;
void foo() {
	std::lock_guard<std::mutex> guard(myMutex);
	myList.push_back(3);
}
```
- With CTAD, can just use `std::lock_guard guard(myMutex);`
- `std::lock_guard`
	- Automatically locks and unlocks a mutex using RAII
	- Mutex is locked on construction and unlocked on destruction, guaranteeing proper mutex release even in case of exceptions
- Common applciations
	- Group mutex and proteted data in a class as private members

### Scoped_lock
- C++17
```cpp
std::scoped_lock guard(myMutex);
```

### How do lock guards work?
```cpp
template <typename Mutex>
class lock_guard {
    Mutex& m_;
public:
    explicit lock_guard(Mutex& m) : m_(m) { m_.lock(); }
    ~lock_guard() { m_.unlock(); }

    lock_guard(const lock_guard&) = delete;
    lock_guard& operator=(const lock_guard&) = delete;
};
```
- RAII
	- Upon construction
		- Pass in a mutex
		- Calls `.lock()` on the mutex
	- Upon **destruction** (automatic, upon going out of scope)
		- Calls `.unlock()` on the mutex

### How does a mutex actually work?
- What happens to threads that try to acquite a mutex that has already been locked by another thread?
	- Thread makes syscall into kernel
	- Kernel puts on wait queue for that mutex, deschedules it (thread stops running entirely and consumes no CPU)
	- When owning thread unlocks, kernel wakes one waiter from the queue and schedules it to run again
- Mutex implementations combine **spin locks** and **sleep locks** 
	- Spin lock: continually run, taking up CPU cycles
	- Combination
		- Spin phase: intially, thread spins for short period
		- If lock not acquired wsithin spin phase, thread transitions to a sleep state
# Structuring code
- Not so simple
	- Pointer or reference can enable racy modification of protected data
> Don't pass pointers/ references to protected data outside the scope of the lock, whether by returning from function, storing in externally visible memory, or passing as arguments to user supplied functions

```cpp
class data_wrapper {
    some_data data;
    std::mutex m;
public:
    template <typename Function>
    void process_data(Function func) {
        std::lock_guard l(m);
        func(data);  // <- this is the problem
    }
};

void foo() {
    data_wrapper x;
    some_data* stolen = nullptr;

    x.process_data([&](some_data& d) {
        stolen = &d;  // captured by the lambda's reference to local
    });

    // mutex is unlocked, stolen points to unprotected data
    stolen->do_something();  // data race
}
```
- The lambda here is dangerous
	- It's taking out the pointer to the protected data and stroign it in `stolen`, allowing the protected data to be modified outside of the mutex
