# `std::shared_mutex`, `std::shared_timed_mutex`
- C++14 for former, C++ 17 for later
	- Latter has more operations
- **Reader-writer** mutex
	- Allows for 2 different kinds of usage
		- Exclusive access by **single** **writer** thread
		- Concurreny access by **multiple** **reader** threads
- Writing operations
	- Lock with `std::lock_guard<std::shared_mutex>` or `std::unique_lock<std::shared_mutex`
- Reading operations
	- Lock with `std::shared_lock<std::shared_mutex>` 
		- Multiple shared threads may have a shared lock on the same `std::shared_mutex` at the same time
	- Constraint
		- If any thread has a shared lock, a thread that tries to acquire an **exclusive lock** will **block** until **all** other threads have relinquished their locks

> Think of a room with whiteboard.
> Anyone can walk in and read whiteboard.
> If someone needs to write on it, all readers have to leave, and no one enters until they're done.

> Only worth then reads vastly outnumber writes.


# `std::unique_lock`
- Relaxes invariants
	- Doesn't always own the mutex it's associated with  
		- 'own' = have it locked right now, and destructor is responsible for unlocking it
		- ie. **Unlike lock_guard, DOESNT lock immediately**
	- Can 
		- **defer locking**
		- unlock **early**
		- be **moved between scopes**
		- do **timed lock attemps**
		- can be used with `conditional_variable`

- Locked automatically on construction
	- **UNLESS** `std::defer_lock` is passed to constructor as an argument
- Can lock and unlock multiple times (different critical regions)

```cpp
// 1. Deferred locking — construct without locking
std::unique_lock<std::mutex> lk(mtx, std::defer_lock);
// ... do some setup ...
lk.lock();    // lock when you're ready

// 2. Early unlock
std::unique_lock<std::mutex> lk(mtx);
...
lk.unlock(); 
// ... expensive non-shared work that doesn't need the lock ...

// 3. Try lock — non-blocking
std::unique_lock<std::mutex> lk(mtx, std::try_to_lock);
if (lk.owns_lock()) {
    // got it
} else {
    // someone else has it, do something else
}

// 4. Timed lock
std::unique_lock<std::mutex> lk(mtx, std::defer_lock);
if (lk.try_lock_for(std::chrono::milliseconds(100))) {
    // acquired within 100ms
}
```

- Unlike lock_guard, doesn't always own the mutex (ie. not always locked)
	- `lock()`, `try_lock()` and `unlock()` member functions
		- Forward to the member functions of the same name of the underlyinh mutex
	- How to know if locked/owned?
		- `owns_lock()` function
	- Why necessary?
		- To ensure that `unlock()` is called correctly in destructor
		- If instance does **own** mutex, `unlock()` **MUST** be called in destructor
		- If it does **not**, `unlock` **must no**t be called
	- Keeping track of this requires a flag
		- Hence `std::unique_lock` is larger and slower than `std::lock_guard`

- Can move between scopes
	- ie. Lock in one scope, **move** to another (retains lock)
```cpp
std::mutex mtx;

std::unique_lock<std::mutex> get_lock() {
    std::unique_lock<std::mutex> lk(mtx);  // locks here
    //...
    return lk; // can also explicitly do std::move(lk), but no need (see below)
}   // lk is moved-from, destructor does nothing

void process() {
    auto lk = get_lock();  // lk now owns the lock
    // ... still holding the lock, do more work ...
}   // unlocks here
```
- Reminder that for returning of local values, the compiler treats it as rvalue automatically
	- Hence no need to return `std::move(lk)`. In fact, doing so will hurt by preventing NRVO

# `std::recusive_mutex`
- Lets the **same thread** lock it **multiple times** without deadlocking
	- Keeps a count and **only actually unlocks when count hits 0**
```cpp
std::recursive_mutex mtx;

void outer() {
    std::lock_guard<std::recursive_mutex> lk(mtx);  // count goes to 1
    inner();  // increments count
}   // count goes to 0, actually unlocks

void inner() {
    std::lock_guard<std::recursive_mutex> lk(mtx);  // count goes to 2
    // ...
}

```
> Should almost never use it. Locking granularity is wrong - locking in both a public method and an internal method it calls

## `std::scoped_lock`


# Problem with locks - Deadlock
- eg. Pair of threads needs to lock pair of mutexes 
	- Each thread has one mutex and is waiting for the other to release
- Solution: always lock 2 mutexes in the same order
	- But what if we can't do this?

## Deadlock soution: `std::lock`
- Can lock 2 or more mutexes **at once** without the risk of deadlock
- eg.
```cpp
std::lock(mutex1, mutex2);
```

Example use case
- Want to swap innter data of 2 `Wrap` objects
- Need to lock both mutexes, do the swap, then unlock
- Use `std::lock()` to lock them
- Then pass `std::adopt_lock` to lock_guards of A and B to **tell them that it's already been locked**
- lock_guards just do `.unlock()`
```cpp
class Wrap {
private:
	Data data_;
	std::mutex m;
public:
	Wrap(Data data): data_(data_) {}
	friend void swap(Wrap& a, Wrap& b) {
		if (&a == &b) return; // Cannot lock twice
		std::lock(a.m, b.m); // Lock both
		std::lock_guard lockA (a.m, std::adopt_lock);
		std::lock_guard lockB (b.m, std::adopt_lock);
		swap(a.data_, b.data_);
		
		// alternatively
		std::scoped_lock guard(a.m, b.m); // does BOTH locking and RAII
	}
}
```
- As an alternative, just do `std::scoped_lock()`
	- Calls `std::lock()` internally on all mutexes
	- Then takes ownership

## Avoiding deadlocks
> Don't wait for another thread if there's a chance it's waiting for you

-  Avoid nested locks - don't acquire a lock if you already hold one
	- Impossible to then get deadlock from lock usage alone, since each thread only ever holds a single lock

- Avoid calling user-supplied code while holding a lock
	- Follow up from above

- Acquire locks in fixed order
	- If we must acquire more than one lock, and can't acquire as a single operation with `std::lock`, then make sure to acquire them in same order in every thread

- Lock hierarchy
	- **thread_local** value representing hierarchy value for the current thread
	- Assign each mutex a numeric level
	- Rule: a thread can **only** lock a mutex whose level is **lower than anything** it currently holds

```cpp
class hierarchical_mutex {
    std::mutex internal_mtx_;
    unsigned long const level_;
    unsigned long previous_level_;  // so we can restore on unlock

    // Each thread tracks the level of its most recently locked hierarchical_mutex
    static thread_local unsigned long this_thread_level_;

    void check_hierarchy() {
        if (level_ >= this_thread_level_) {
            throw std::logic_error("lock hierarchy violated");
        }
    }

public:
    explicit hierarchical_mutex(unsigned long level)
        : level_(level), previous_level_(0) {}

    void lock() {
        check_hierarchy();              // throws if violation
        internal_mtx_.lock();           // actually lock the mutex
        previous_level_ = this_thread_level_;  // save old level
        this_thread_level_ = level_;    // update thread's current level
    }

    void unlock() {
        if (this_thread_level_ != level_) {
            throw std::logic_error("unlocking in wrong order");
        }
        this_thread_level_ = previous_level_;  // restore old level
        internal_mtx_.unlock();
    }

    bool try_lock() {
        check_hierarchy();
        if (!internal_mtx_.try_lock()) return false;
        previous_level_ = this_thread_level_;
        this_thread_level_ = level_;
        return true;
    }
};

thread_local unsigned long
    hierarchical_mutex::this_thread_level_ = ULONG_MAX;  // start at max so any
```

```cpp
hierarchical_mutex high_level(10000);
hierarchical_mutex mid_level(5000);
hierarchical_mutex low_level(1000);

void correct() {
    std::lock_guard<hierarchical_mutex> lk1(high_level);   // 10000, ok
    std::lock_guard<hierarchical_mutex> lk2(mid_level);    // 5000 < 10000, ok
    std::lock_guard<hierarchical_mutex> lk3(low_level);    // 1000 < 5000, ok
}
```
- The **static thread_local** variable `this_thread_level` inside the custom mutex class keeps track of the current level
	- thread_local means each thread has its own copy of this_thread_level_. 
	- After lock 1, this variable has reduced, meaning we can only then lock lower level mutexes.
	- Only after lock1 unlocks, can we go back to locking high level mutexes, and so on.
- `previous_level_`  is stored in the mutex itself, creating a stack-like chain. 
	- When you lock three mutexes at levels 10000 → 5000 → 1000, and then unlock 1000, it restores this_thread_level_ to 5000. Unlock 5000, it restores to 10000. 
	- The unlocks must happen in reverse order, matching RAII destructor order.
 - It works with lock_guard because hierarchical_mutex provides lock(), unlock(), and try_lock() — that's the Lockable concept that lock_guard requires. The hierarchy check is invisible to the caller.

Why it works?
- Deadlock requires a cycle
- With hierarchical checks, there never is a cycle as it means going up
	- Every thread locks in the same top-down order
	- The following cannot happen
```cpp
Thread 1: lock A → lock B 
Thread 2: lock B → lock A ← opposite order, deadlock possible
```

# Locking at appropriate granularity
- Where possible, lock a mutex **only while accessing the shared data**
- Try do do any processing of the dat **outside the lock**
	- Especially don't do any file I/O while holding the lock
- `std::unique_lock` useful here 
	- can call `unlock()` when code no longer needs access to shared data
	- `lock()` again when access is required later
- Lock should only be held for the minimum possible time needed to perform required operations
