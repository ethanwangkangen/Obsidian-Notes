# Waiting
- How does waiting thread know when to wake up?
# Naive checking
- Have the waiting thread check a flag that's protected by a mutex
```cpp
bool flag;
std::mutex m;
void checkFlag() {
	std::unique_lock lock(m); // lock the flag to check
	while (!flag) {
		lock.unlock(); // unlock it again so other threads can check
		std::this_thread::sleep_for(std:;chrono::milliseconds(100));
		lock.lock(); // lock it again to repeat the loop
	}
}
```

# Condition variables
- `#include <condition_variable>`
## `std::condition_variable`, `std::condition_variable_any`
- Former is limited to working with `std::mutex`
- Latter can work with anything that meets the minimal criteria for being **mutex-like**

```cpp
std::mutex mtx;
std::queue<DataChunk> workQueue;
std::condition_variable cv;

void producer() {
	while (moreDataToPrepare()) {
		DataChunk const data = prepareData();
		{
			// lock
			std::lock_guard<std::mutex> lock(mtx);
			workQueue.push(data);
			// unlock
		}
		cv.notify_one();
	}
}

void consumer() {
	while (true) {
		// lock
		std::unique_lock<std::mutex> lock(mtx);
		
		cv.wait(lock, [&] {
			return !workQueue.empty();
		});
		
		DataChunk data = workQueue.front();
		workQueue.pop();
		
		// unlock
		lock.unlock();
		process(data);
	}
}

```
- Producer
	- Lock queue
	- Push data in
	- Unlock queue (automatic with RAII)
	- Notify 
		- After unlocking, so that if the waiting thread wakes immediately, don't have to block again, waiting to unlock the mutex
- Consumer
	- Lock mutex
	- Wait for notification
		 - If lambda not satisfied (ie. queue is empty), wait() unlocks the mutex and puts the thread in blocked/waiting state
		 - When the conditional variable is notified by call to `notify_one()` in producer
			 - the thread unblocks,
			 - re-locks, 
			 - checks condition again, 
			 - returns from wait() with mutex still locked if condition satisfied
	- Take out data
	- Unlock
	- Then process

- We need `unique_lock()` for the consumer since we need to unlock and lock again
	- Waiting thread must unlock the mutex while it's waiting and lock it again afterwards

# std::semaphore
- `#include <semaphore>`
- `std::counting_semaphore<N> sem(n)`
	- N is total, n is starting number
	- n <= N
	- n < N -> used when we want some threads to wait.
		- Remember that semaphores don't have ownership, meaning that a **thread can release without acquiring**
- An **atomic** counter with 2 operations
	- `acquire()` -> decrement count
		- If count is 0, waits until count > 0
	- `release()` -> increment count
- Think of it as a parking lot with N lots
	- Each car entering acquires spots 
	- Each car leaving releases
	- If spots == 0, incoming cars wait
- Difference from mutex
	- Mutex is owned by a thread. Semaphore has **no ownership** - any thread can release
- `std::binary_semaphore`
	- Just an alias for `std::counting_semaphore<1>`

```cpp
#include <semaphore>

// Counting semaphore — counter can go up to some max
std::counting_semaphore<10> sem(3);  // max 10, starts at 3

sem.acquire();   // counter: 3 → 2  (blocks if counter == 0)
sem.acquire();   // counter: 2 → 1
sem.acquire();   // counter: 1 → 0
sem.acquire();   // BLOCKS — counter is 0, waits for a release

sem.release();   // counter: 0 → 1, wakes one blocked thread
sem.release(3);  // counter: 1 → 4, can release multiple at once
```

- Application of semaphore in SPSC
```cpp
/* Globals */
getcracked::queue buffer;
std::binary_semaphore buffer_semaphore{0};
std::mutex buffer_mutex;

/* Consumer thread */
void consumer() {
    std::unique_lock<std::mutex> lock(buffer_mutex); 
    if (buffer.is_empty()) {
	    lock.unlock();
        buffer_semaphore.acquire();
        lock.lock();
    }
    // consumer consumes resources
}

/* Producer thread */
void producer() {
    std::lock_guard<std::mutex> lock(buffer_mutex);
    // producer produces resources
    buffer_semaphore.release();
}
```
- Why did we need to use unique lock, and unlock first in the consumer?
	- If not, deadlock
	- eg. if we had just used a normal lock guard,
		- Imagine if consumer runs first, an thus semaphore count is still at 0
		- Consumer locks buffer_mutex -> buffer empty -> acquires semaphores -> BLOCKS (waits, since count is 0)
		- Producer tries to lock buffer_mutex -> BLOCKS (consumer holds it)
	- To solve, we need to let the producer acquire the mutex, so `release()` can be called

- What's the point of the semaphore here?
	- Used as a **signal** to tell consumer that a new item has been placed in the buffer

# `std::latch`
- **Single-use countdown barrier**
	- Set a count, threads decrement it.
	- Threads **waiting** on it **wake up** when count hits **zero**
		- When count hits 0, stays there forever
- `arrive_and_wait()`
	- Decrement count and block until count == 0
- `count_down()`
	- Just decrement. Don't wait.
- `wait()`
	- Wait for count to hit 0.
```cpp
#include <latch>

std::latch startGate(3);  // count = 3

// Three worker threads
void worker(int id) {
    prepareWork(id);
    startGate.arrive_and_wait();  
    doWork(id);
}
```

# Thread safe queue (textbook version)
```cpp
#include <queue>
#include <memory>
#include <mutex>
#include <condition_variable>

template<typename T>
class threadsafe_queue {
private:
    mutable std::mutex mut;              // (1) must be mutable
    std::queue<T> data_queue;
    std::condition_variable data_cond;

public:
    threadsafe_queue() {}

    threadsafe_queue(threadsafe_queue const& other) {
        std::lock_guard<std::mutex> lk(other.mut);
        data_queue = other.data_queue;
    }

    void push(T new_value) {
        std::lock_guard<std::mutex> lk(mut);
        data_queue.push(new_value);
        data_cond.notify_one();          // (2) notify after pushing
    }

    void wait_and_pop(T& value) {
        std::unique_lock<std::mutex> lk(mut);
        data_cond.wait(lk, [this]{ return !data_queue.empty(); });
        value = data_queue.front();
        data_queue.pop();
    }

    std::shared_ptr<T> wait_and_pop() {
        std::unique_lock<std::mutex> lk(mut);
        data_cond.wait(lk, [this]{ return !data_queue.empty(); });
        std::shared_ptr<T> res(std::make_shared<T>(data_queue.front()));
        data_queue.pop();
        return res;
    }

    bool try_pop(T& value) {
        std::lock_guard<std::mutex> lk(mut);
        if (data_queue.empty())
            return false;
        value = data_queue.front();
        data_queue.pop();
        return true;
    }

    std::shared_ptr<T> try_pop() {
        std::lock_guard<std::mutex> lk(mut);
        if (data_queue.empty())
            return std::shared_ptr<T>();
        std::shared_ptr<T> res(std::make_shared<T>(data_queue.front()));
        data_queue.pop();
        return res;
    }

    bool empty() const {
        std::lock_guard<std::mutex> lk(mut);
        return data_queue.empty();
    }
};   
```
Things to note
- Mutex must be mutable
	- `empty()` is a const function, but locking a mutex modifies it so need `mutable` to let you lock it inside const methods
- `unique_lock` for waiting, `lock_guard` for everything else
	- `condition_variable::wait` must be able to unlock and relock the mutex while it sleeps, requires `unique_lock` to do that
- Predicate in `wait` handling spurious wakeups
	- `data_cond.wait(lk, [this]{ return !data_queue.empty()});`

# Thread safe queue (practice version)
```cpp
#include <condition_variable>

template <typename T>
struct DataWrapper
{
    T data{ };
    bool is_last_chunk{ false };
};

template <typename T, typename Callback>
class SPSC
{
public:
    SPSC(Callback callback) : 
        callback_(callback)
    {
        consumerThread = std::thread(&SPSC::Consume, this);
    }

    void PushWork(const DataWrapper<T>& wrapper)
    {
        std::unique_lock lock(mtx);
        q.push(wrapper);
        cv.notify_one();
    }

    ~SPSC() 
    {
        PushWork({{}, true});
        consumerThread.join();
    }

private:
    std::mutex mtx;
    std::queue<DataWrapper<T>> q;
    std::condition_variable cv;
    Callback callback_;
    std::thread consumerThread;

    void Consume()
    {
        while (true) {
            // lock the mutex
            std::unique_lock lock(mtx);

            // wait until it's not empty
            cv.wait(lock, [&](){
                return !q.empty();
            });

            // check queue
            auto wrapper = q.front();
            q.pop();

            // unlock, no longer needed
            lock.unlock();
            // process
            callback_(wrapper.data);
            // if last element, exit loop
            if (wrapper.is_last_chunk) break;
        }
        
    }
};
```
Things to note
- Variables
	- Mutex, queue, cv, **callback function lambda**, **consumer thread**
- Producer thread
	- Lock, push and notify. simple
- Consumer thread
	- Is a **member variable** of the queue class
	- Runs in a **loop**
		- Breaks on last element seen
	- Wait, then check **predicate** (which is that queue is not empty)
	- Remember to **unlock before processing**
- Callback lambda 
	- Is passed in from the constructor
	- Has to be a **member** of the class
	- Has to be **member initialised**, cannot do `thread_ = ...` since lambdas have deleted default constructor and deleted copy assignment
- Destructor
	- Remember to **join the thread** if not UB!!!
# Reader-Writer lock
```cpp
namespace getcracked 
{
    class ReaderWriterLock
    {
    public:

        // only if no more writers and no pending writers
        unsigned ReaderLock()
        {
            std::unique_lock lock(mutex_);
            readers_ready.wait(lock, [&] {
                return currentWriters == 0 and pendingWriters == 0; 
            });
            return currentReaders++;
        }

        // if no readers left, notify pending writer
        unsigned ReaderUnlock() 
        { 
            std::unique_lock lock(mutex_);
            currentReaders--;
            if (currentReaders == 0 and pendingWriters != 0) writers_ready.notify_one();
            return currentReaders;
        }

        // only if no current writer
        void WriterLock() 
        { 
            std::unique_lock lock(mutex_);
            pendingWriters++;
            writers_ready.wait(lock, [&] {
                return currentWriters == 0 and currentReaders == 0;
            });
            pendingWriters--;
            currentWriters++;
        }

        void WriterUnlock()
        { 
            std::unique_lock lock(mutex_);
            --currentWriters;
            if (pendingWriters) {
                writers_ready.notify_one(); 
                // Note: still correct if we notify_all() writers, since ulimately the predicate ensures that only one of them gets through
            } else {
                readers_ready.notify_all();
            }
        }

    private:
        std::mutex mutex_;
        std::condition_variable readers_ready;
        std::condition_variable writers_ready;
        unsigned currentReaders{};
        unsigned currentWriters{}; // should be max one
        unsigned pendingWriters{};
    };
}
```
- Things to note
	- Start with variables
		- Number of readers
		- Number of writers
		- Number of pending writers
	- Then start with the condition variables to notify threads
		- One for the writer threads
		- One for the reader threads
	- All functions start with `unique_lock` to lock the mutex
		- `unique_lock` instead of `lock_guard` because `.wait()` needs it
	- Locking
		- **Wait** until ok, then mutate
			- Note: the predicate is **not** scheduler, that is controlled by the OS.
			- Eg. if `notify_all()` wakes 5 threads and 3 have true predicates, which of those 3 grabs mutex is the OS' decision.
		- Who gets woken up is controlled by `notify_one/notify_all`
			- Is just a hint to recheck the predicate
		- **Don't** require `notify` to be called in order for the predicate to be true! This is not `acquire/release()` on a semaphore.
	- Unlocking
		- Mute then notify
