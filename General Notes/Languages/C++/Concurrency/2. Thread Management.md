# Creating thread
```cpp
#include <thread>
// function
void f() {...}
std::thread my_thread(f);

// lambda
std::thread ([]() {
	f();
});
```
- f can be anything with function call operator
	- Function
	- Functor with `()` callable
	- Lambda

Be careful with most vexing parse! For passing a temporary functor in.
- `std::thread my_thread(background_task());`
	- Most vexing parse!

### Can pass in arguments to threads as well
```cpp
std::thread my_thread(f, 5);
```
- Number of arguments
	- Must match exactly
	- Add the implicit `this` pointer as first argument if passing in member function pointer

- Internally, 
	- std::thread uses forwarding references, so we can pass in lvalues or rvalues.
		- This determines **how** the stored object is constructed (*copy vs move*)
	- The thread then reconstructs the object via **move/copy** constructor.
		- Because of the **decaying**
		- Hence the need for std::ref, because even if we pass in referneces, objects are recreated

- Arguments are **copied by default**, even if the function takes in a reference
```cpp
void process(vector<int>& data);
vector<int> v {1,2,3};
std::thread t(process, v); // DOES NOT COMPILE

std::thread t(process, std::ref(v)); // wraps v in reference wrapps
```
- The former fails to compile, as 
	- std::thread stores a **copy** of v internally
		- Copy is stored as **rvalue**
	- Non-const reference `data` cannot bind to rvalue
	- Hence, need to wrap in a `std::reference_wrapper<>` . Copyable, implicitly converts to reference at call site

- Decay copy semantics
	- Each argument is `decay_copy`d -> array decay to poitner, function decay to function pointer, top level cv-qualifiers and references stripped
	- Hence, passing raw array gives pointer (potentially dangling if array is local), and passing reference doesn't just work (decayed type is a value, not ref)
	- eg. the **const char* dangling reference trap** below
```cpp
void greet(const std::string& name);

void spawn() {
    char buf[64];
    snprintf(buf, sizeof(buf), "Ethan");
    
    // MISTAKE!
    std::thread t(greet, buf); 
    
    // correct:
    std::thread (greet, std::string(buf));
    
    t.detach();
}  // buf is destroyed here
```
- thread copies the **pointer**, not the actual buffer contents
- Solve by explicitly constructing `std::string` in the calling thread

- Move only types
	- Move them in
	- `std::thread` internal storage will move-construct from the rvalue
```cpp
void consume(std::unique_ptr<Widget> w);

auto p = std::make_unique<Widget>();
std::thread t(consume, std::move(p));  // must move — can't copy unique_ptr
```

- Member function pointers
	- Remember to pass the `this` pointer in
	- The object itself must outlive the thread
```cpp
class OrderBook {
public:
    void process_feed(int feed_id);
};

OrderBook book;
std::thread t(&OrderBook::process_feed, &book, 42);
// first argument is the implicit *this
```

# Summary of argument passing to thread
**1. By value — just pass it.**
```cpp
std::thread t(f, 42);   // 42 copied into storage, passed to f
```

**2. By reference — you must wrap in `std::ref` (or `std::cref`).**
```cpp
int n = 5;
std::thread t(f, std::ref(n));   // f(int&) now sees the real n
```

- Without `std::ref`, the thread copies `n` and the function either fails to compile (for `T&`) or silently operates on the copy (for `const T&`). `std::ref` produces a `reference_wrapper` that survives the decay-copy and converts back to `T&` at the call site. 
- **You own the lifetime** — `n` must outlive the thread.

**3. Member functions — pointer-to-member, then the object, then args.**
```cpp
std::thread t(&Widget::member, &w, 10);   // calls w.member(10)
```

- Second argument is the "this": a pointer (`&w`), a reference via `std::ref(w)`, or a smart pointer. 
- If you pass `w` by value it copies the object and the method acts on the copy.

**4. Move-only arguments — `std::move` them in.**
```cpp
auto p = std::make_unique<int>(99);
std::thread t(consume, std::move(p));   // ownership transferred into the thread
```

- Because args are moved into storage, move-only types (`unique_ptr`, `future`, `packaged_task`) work — but you must explicitly `std::move`, and `p` is null afterward in the caller.

**5. The dangling-conversion hazard (Williams' big one).** 
- If a parameter type needs _conversion_ (e.g. `char const*` → `std::string`), the conversion happens **inside the new thread**, using the copied pointer — which may dangle by then:

```cpp
// DANGER: buf is a local; thread copies the POINTER, converts later, buf may be gone
std::thread t(f, 3, buf);
// SAFE: force the conversion NOW, in the calling thread
std::thread t(f, 3, std::string(buf));
```

>Rule: **convert to the final type yourself before passing** whenever an implicit conversion from a pointer/temporary is involved.

# detach/join
- Detach
	- Leave the thread running on its own, don't wait for it
- Join
	- Wait for thread to finish
	- Cleans up storage associated with the thread
		- `std::thread` object no longer associated with this now-completed thread
		- The `std::thread` object is no longer joinable, and `joinable()` returns false
- Note: if we don't decide one of these before the std::thread object is destroyed, the program is **terminated**
	- **MUST call one of these functions!**

- Be careful of **dangling references/pointers** when using detach
	- eg.
```cpp
struct func {
	int& i_; // captured by reference
	func(int& i) : i_(i) {}
	
	void operator() ()
	{
		do_something(i);
	}
};

void oops () {
	int i {5};
	func myFunc(i); // pass in reference
	std::thread myThread(myFunc);
	myThread.detach; // DANGEROUS! Dangling reference
}
```
- Thread detaches
	- i is captured by reference but thread may still be running after oops() terminates and i goes out of scope

### Considering exceptions when waiting
- In general, if calling `.join()` in **non-exceptional** case, we ALSO need to call it in the case of exceptions
```cpp
std::thread t(myFunc);
try {
	do_something_in_current_thread();
} catch (...) {
	t.join();
	throw;
}
t.join(); // here too
```
- Alternative
	- Instantiate a custom thread guard object, that takes in a thread by reference and calls the `.join()` on it in its **destructor**
	- Delete copy construtor and assignment to ensure not auto provided by compiler
```cpp
class thread_guard
{
private:
	std::thread& t_;
public:
	thread_guard(std::thread t) : t_(t) {}
	thread_guard(const thread_guard&) = delete;
	thread_guard& operator=(const thread_guard&) = delete;
	~thread_guard() {
		if (t.joinable()) t.join();
	}
};
```
- Can consider having the thread_guard **take ownership** of the thread and **move it in** too
- `std::thread t_;`
- `thread_guard(std::thread t) : t_(std::move(t)) {}`
  
### Threads in background
- Call `.detach()`
	- eg. Opening a new documnet. The thread handling the request doesn't need to care about waiting for that other thread to finish
