# Types of template parameters
## 1. Type template parameter

```cpp
template <typename T> //function generated
T add(T a, T b) {
    return a + b;
}

add<int> (a,b);
```

- The first time a function call is made, function instantiation happens
	- T is replaced with the specific type
- How deduction works
	- Base type is deduced first `T`
	- Then parameter structure is applied `T&, T*, T&&`

### Deduction Contexts
- Match argument to pattern
- Deduce base type only (T)
- Remove reference (unless T&), remove top-level const (unless reference)
- Apply & or && from function signature
- Apply reference collapsing if needed
#### 1. By value
```cpp
template <typename T>  
void f(T x);
```
- References removed
- Top-level const removed
	- Since a copy is created anyway

#### 2. By lvalue reference
```cpp
template <typename T>
void f(T& x);
```
- Must bind to **lvalue**
- **const** is preserved
	- eg. `const int a = 10; f(a);` 
		- `a has to be of type const int&, so T is const int`
- Reference is **not** part of T

#### 3. By const lvalue reference
```cpp
template <typename T>
void f(const T& x);
```
- Can bind to **lvalue AND rvalue**
	- However, rvalue treated as const, **cannot move from them**
- So use case is only if need to **view** the value
- T is deduced without reference
- const is added regardless of x type
- **const** is part of parameter, not T
	- eg. T = int, parameter = const int &

#### 4. Forwarding reference
```cpp
template <typename T>
void f(T&& x);
```
- Basically, whether input is lvalue or rvalue, the final parameter will be the corresponding reference type.
- Why necessary?
	- We might want to preserve an rvalue's status as an rvalue, and likewise foe lvalue
	- Because if its an lvalue, **should not steal from it -> should copy**
	- If it's rvalue, **allowed to move from it**
- So `T&&` allows us to preserve lvalue vs rvalue, enables move semantics and perfect forwarding

| input  | T      | parameter                  |
| ------ | ------ | -------------------------- |
| lvalue | `int&` | `int&` (lvalue reference)  |
| rvalue | `int`  | `int&&` (rvalue reference) |
Reference collapsing rule:
```
& + & = &  
& + && = &  
&& + & = &  
&& + && = &&
```
- If there is any `&`, the result is `&`

## 2. Non-type template parameter
- Usually used for passing **constexpr** values to functions so they can be used in contexts that require constexpr
	- **Because function parameters cannot be constexpr**
```cpp
template <int N> // object generated
class Array {
public:
    int data[N];
};
Array<100> b;


template <int N>
void printNTimes() {
    for (int i = 0; i < N; i++)
        std::cout << "Hello\n";
}
printNTimes<3>();


template <double D> // requires C++20 for floating point non-type parameters
double getSqrt() {
    static_assert(D >= 0.0, "getSqrt(): D must be non-negative");

    if constexpr (D >= 0) // ignore the constexpr here for this example
        return std::sqrt(D); // strangely, std::sqrt isn't a constexpr function (until C++26)
    return 0.0;
}
```
- Why constexpr parameters better?
	- Evaluation at compile time -> save on runtime
	- Can use `static_assert()`, `if constexpr()` etc.

## 3. Template template parameter
- Template parameter represents a template
```cpp
template <template <typename> class Container> //object generated
class Wrapper {
    Container<int> data;
};
Wrapper<std::vector> w;
```


# Templates in multiple files
- Cannot define in add.cpp and forward declare + use in main.cpp file.
	- Because main.cpp doesn't see definition, can't instantiate the function
- Solution: put **all template code in header file**
	- Does not violate ODR, since ODR says that types, templates, inline functions and inline variables can have identical definitions in different files


# Argument deduction
- `add<> (a,b);`
	- Templates only
- `add (a,b);`
	- Both non-template and template considered.
	- **Non-template preferred**
		- But only prefer non-template under equal footing. Normal rules for precedence still apply first
	- Prefer using this

### Argument deduction - multiple template types
```cpp
T max(T x, T y);
max(1.5, 3) // Try to use argument deduction here. wont' compile!!
```
- Here, T can only represent a single type
	- The compiler won't generate `max<double>` then convert int to a double.
	- Type conversion is done only when **resolving function overloads**, NOT when performing **type deductions**
- Solutions
	- Static convert to same type
		- `max(static_cast<double>(2), 3.5)
	- Don't use type deduction
		- `max<double>(2, 3.5)`
		- Type convert will work here
	- Use different parameter types
		- `auto max (T x, U y)`
		- To forward declare this, use **std::common_type_t**
			- `auto max(T x, U y) -> std::common_type_t<T, U>;`
# Other notes
### Static variable
 - If a template function has a **static variable**, differently instantiated functions **do not share** the same static variable
# Abbreviated function templates (C++20)
- Syntactic sugar for function templates using the `auto` parameter
```cpp
// Instead of writing 
template <typename T>
void f(T x) {
}

// can write
void f (auto x) {
}
```
- With references
```cpp
void f(auto& x);

// becomes
template <typename T>
void f(T& x); // lvalues can bind, const correctness preserved in deduction
```

- With `auto&&`
```cpp
void f(auto&& x);

// becomes
template <typename T>
void f(T&& x);
```
