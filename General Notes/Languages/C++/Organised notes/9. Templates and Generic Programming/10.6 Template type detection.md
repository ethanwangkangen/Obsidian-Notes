```cpp
template <typename T>
void f(ParamType param); // ParamType = T&, const T&, T&&
```

# Case 1: ParamType is reference/pointer, but not universal reference
```cpp
int x{};
const int cx = x;
const int& rx = x;

f(x);
f(cx);
f(rx);
```

- The compiler tries to make the call argument match the **parameter** type pattern **first**, THEN deduces T accordingly
	- **Ignore the reference part of the expression parsed in**
	- Then try to match Param, to determine T
     
| ParamType -> | T&        |            | const T& |            |
| ------------ | --------- | ---------- | -------- | ---------- |
|              | T         | Param      | T        | Param      |
| x            | int       | int&       | int      | const int& |
| cx (&)       | const int | const int& | int      | const int& |
| rx (&)       | const int | const int& | int      | const int& |
- Note: works the same way for rvalues too (for const T&), even though not shown
- rvalues not allowed for T&

- Works the same way for pointers.
```cpp
int x = 27; // as before
const int *px = &x; // px is a ptr to x as a const in
```

| ParamType-> | T*        |            |
| ----------- | --------- | ---------- |
|             | T         | param      |
| &x          | int       | int*       |
| px          | const int | const int* |

# Case 2: ParamType is a Universal Reference

| ParamType->              | T&&        |            |
| ------------------------ | ---------- | ---------- |
|                          | T          | Param      |
| x (becomes lvalue, so &) | int&       | int&       |
| cx (&)                   | const int& | const int& |
| rx (&)                   | const int& | const int& |
| 27                       | int        | int&&      |
- If lvalue passed in, **both T and ParamType** deduced to be **lvalue referenes**
	- **lvalue is converted into an lvalue reference**
	- Special case! Normally lvalues don't just convert to lvalue references.
- If rvalue passed in, Case 1 rules apply.
- Downside
	- T can end up being a reference type, which may not be intuitive
### Reference collapsing rules
- For determining the **parameter**
- If lvalue is passed in, first change it into an lvalue reference
& + & → &
&  + && → &
&& + & → &
&& + && → && (rare)
# Case 3: ParamType is neither pointer nor reference
- Pass by value
- If expr type is reference, ignore the reference part
- If after that, expr is const, ignore that too
	- Be careful with which const is dropped! See below
```cpp
template <typename T>
void f(T param); // param is still passed by value 

const char* const ptr = "Fun with pointers"; // ptr is const pointer to const object 

f(ptr);
```
- The `ptr` is passed by value
	- But it's still a pointer to `const char*`
	- The constness of what `ptr` points to is preserved
		- But the constness of the `ptr` itself is ignored

### Special rule for universal reference taking lvalues
- When lvalue passed into universal reference, it becomes an lvalue reference. Then collapsing rules apply
# Array arguments
Pointer decay
- Remember that 
	- `const char name[] = "ethan"` decays to
	- `const char *`
- And that function parameters will be as `const char *`

- Function parameters can't declare parameters that are truly arrays
	- But can declare parameters that are **references** to arrays
```cpp
template <typename T>
void f(T& param);

const char name[] = "ethan";
f(name);
```
- `T` type becomes the array type!!
	- ie. `const char[6]`
- param type is `const (&) char[6]`

- The same applies to **function arguments**
	- Function types decay into pointers
# To remember
```
During template type deduction, arguments that are references are treated as non-references, i.e., their reference-ness is ignored. 

When deducing types for universal reference parameters, lvalue arguments get special treatment. 

When deducing types for by-value parameters, const and/or volatile argu‐ ments are treated as non-const and non-volatile. 

During template type deduction, arguments that are array or function names decay to pointers, unless they’re used to initialize references.
```

```cpp
#include <iostream>
#include <type_traits>

template <typename T>
void check() {
    std::cout << std::is_same_v<T, int> << " "
              << std::is_same_v<T, int&> << " "
              << std::is_same_v<T, int&&> << "\n";
}

template <typename T>
void forward_to_check(T&& x) {
    check<T>();
    check<decltype(x)>();
}

int main() {
    int a = 5;
    forward_to_check(a);
    std::cout << "---\n";
    forward_to_check(42);
}
```
- First case
	- lvalue passed in
	- T&& deduced as int&, x deduced as int&
		- T is int&, decltype(x) is int&
	- 0 1 0
	- 0 1 0
- Second case
	- rvalue passed in
	- T&& deduced as int&&, x deduced as int&&
		- T is int, decltype(x) is int&&
	- 1 0 0
	- 0 0 1

### For forwarding references
- Deduce T first!
	- If lvalue int passed in, T is `int&`
	- If rvalue int passed in, T is `int`
	- As simple as this
- Then apply collapsing rules `T&&` to get the parameter type
- So the chain is
	- `value category of argument → T → collapse T&& → parameter type = decltype(x).`

> Always start from T. derive everything else from it

```cpp
f(T&&  x)  →  T can be a reference (special rule)
f(T&   x)  →  T is never a reference
f(T    x)  →  T is never a reference, also strips const
```

#### Don't get confused
- References are transparent in expressions
- The compiler sees the value category and the underlying type, not the declared reference-ness of the variable
```cpp
int a = 5;
int& ref = a;
int&& rv = 42;

forward_to_check(a);     // lvalue int → T = int&
forward_to_check(ref);   // lvalue int → T = int&   (same)
forward_to_check(rv);    // lvalue int → T = int&   (same)
```
