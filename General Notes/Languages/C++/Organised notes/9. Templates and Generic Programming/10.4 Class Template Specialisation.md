Eg.
```cpp
template <typename T>
class Storage8 {
private:
    T m_array[8];

public:
    void set(int index, const T& value) {
        m_array[index] = value;
    }

    const T& get(int index) const {
        return m_array[index];
    }
};
```
- `Storage8<bool>` will end up being inefficient
	- All variables must have an address, and CPU can't address if less than a byte, so all variables must be at least one byte in size
	- But `bool` only needs 1 byte

# Class Template specialisation
```cpp
//... (the code on top)

template <> // no parameters
class Storage8<bool> { // specializing Storage8 for bool
private:
    std::uint8_t m_data{};

public:
    void set(int index, bool value) {
        // Figure out which bit we're setting/unsetting
        // This will put a 1 in the bit we're interested in turning on/off
        auto mask{ 1 << index };

        if (value)  // If we're setting a bit
            m_data |= mask;   // use bitwise-or to turn that bit on
        else  // if we're turning a bit off
            m_data &= ~mask;  // bitwise-and the inverse mask to turn that bit off
	}

    bool get(int index) {
        // Figure out which bit we're getting
        auto mask{ 1 << index };
        // bitwise-and to get the value of the bit we're interested in
        // Then implicit cast to boolean
        return (m_data & mask);
    }
};
```
- Specialised class template starts off with `template<>`
- When we instantiated any `Storage<T>` where T is not a bool, it will create a new version
	- But when T is bool, it will use this version here

## Specialising member functions
- What if we just want to customise a **member function** to be used for a specific type, but retain the rest of the class?
	- Define it with the specified type outside the class

```cpp
template<>
void Storage<double>::print() {
    std::cout << std::scientific << m_value << '\n';
}
```
- Explicit function specialisation
	- Not implicitly inline, make inline if put in header file

### Don't get confused!
- Above, we are **specialising** the member function.
- This is different from **DEFINING** a member function of a fully specialised class
	- See example below
```cpp
template <typename T>
class S;

template<> 
class S<char**> { // this is a FULLY specialised class, == normal class 
    public:
        void print() const;
};

void S<char**>::print() const { };
```


## Where to define class template specialisations
- Compiler must see full definition of both the non-specialised class
	- So just define it in the same header file, just below the non-specialised (generic) class
- If specialisation only required in a **single** translation unit
	- Then just put it there
	- Other translation units will not see definition hence continue to use the non-specialised version
- Be wary of putting specialisation in its own header file then include the header in any translation unit where it's needed
	- Dangerous to change behaviour based on presence/absence of a header file


# Template instantiation
- For class templates, **member functions** that are **unused** will **not be instantiated**
- Has implications
	- Eg. template member functions cannot be **virtual**

### Member function templates cannot be virtual
- Virtual mechanism creates a table of fixed size with one entry per virtual method. 
	 - However, the **number of instantiations** of **virtual method templates** is **not fixed** until the entire program has been translated
		 - Think of the case of member function that is defined and unused, it will be **discarded**
	- In contrast, ordinary methods like **virtual destructor** are viable since there will always be one per instance
```cpp
template <typename T>
class Dynamic {
public:
    virtual ~Dynamic(); // Allowed

    template <typename U>
    virtual void copy(const U&); // Not allowed
};
```
- Don't get confused
	- A class template can have virtual member functions (eg. the destructor)
	- But **member function templates** cannot be made virtual (a template function inside a class)

```cpp
// class template with a virtual function — OK
template <typename T>          // ← template here
struct A {
    virtual void f(T);         // ← virtual function, but not itself a template
};

// virtual member function template — NOT OK
struct B {
    template <typename T>      // ← template here
    virtual void f(T);         // ← virtual AND templated
};
```


# Partial template specialisation
- Allows us to specialise **classes** (but not individual functions!) where some (not all) of template parameters have been explicitly defined

- Which is chosen?
	- Choose the "most specialised one"

1) If only one specialization matches the template arguments, that specialization is used
2) If more than one specialization matches, partial order rules are used to determine which specialization is more specialized. The most specialized specialization is used, if it is unique (if it is not unique, the program cannot be compiled)
3) If no specializations match, the primary template is used

- What does 'more specialised' mean?
	- One is a **strict subset** of the other

Ambiguous example
```cpp
template <typename T1, typename T2>
class Obj {
};

// Spec 1
template <typename T>
class Obj<T, T> {
};

// Spec 2
template <typename T>
class Obj<T, int> {
};

int main() {
    Obj<int, int> obj;
}
```
- Ambiguous
	- Spec 2 is NOT a strict subset of spec 1, different dimensions altogether
- Hence both specialisations equally valid

More specialised 
```cpp
template <int I, int J, class T>
struct X {};                                    // primary

// Spec 1
template <int I, int J>
struct X<I, J, int> { static const int s = 1; }; // #1

// Spec 2
template <int I>
struct X<I, I, int> { static const int s = 2; }; // #2

int main() {
    X<2, 2, int> x;  // both #1 and #2 match
    std::cout << x.s; // prints 2
}
```
- Spec 2 is a **strict subset** of Spec 1, hence more specialised.


## Partial specialisation - multiple parameters

```cpp
template <typename T, typename U>  
class Pair {  
public:  
    static void print() {  
        std::cout << "generic pair\n";  
    }  
};

//Partial:
template <typename T>  
class Pair<T, T> {  
public:  
    static void print() {  
        std::cout << "same types\n";  
    }  
};

Pair<int, double>::print(); // generic  
Pair<int, int>::print(); // specialised
```


## Partial specialisation for pointers
```cpp
template <typename T>
class Storage {// This is our primary template class (same as previous)
}


template <typename T> // we still have a type template parameter
class Storage<T*> {// This is partially specialized for T*
private:
    T* m_value {};
public:
    Storage(T* value)
      : m_value { value }
    {
    }

    void print();
};

template <typename T>
void Storage<T*>::print() { // This is a non-specialized function of partially specialized class Storage<T*>
    if (m_value)
        std::cout << std::scientific << *m_value << '\n';
}
```
- Note: type template parameter is still defined as `T` and not `T*`
	- T is the non-pointer type, so we have to use `T*` anywhere we want a pointer to T

### Ownership and lifetime issues
- If object that `m_value` points to is destroyed, the `Storage<T*>` is left dangling
- Best to avoid use of `Storage<T*>` altogether.
```cpp
template <typename T>
class Storage {
    // Make sure T isn't a pointer or a std::nullptr_t
    static_assert(!std::is_pointer_v<T> && !std::is_null_pointer_v<T>, "Storage<T*> and Storage<nullptr> disallowed");
```
- Or, allocate the object on the heap, and have `Storage<T*>` hold a `unique_ptr<T>` to automatically deallocate when Storage is destroyed
