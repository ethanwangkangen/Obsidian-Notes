
# Function template overloading
(Not doing template specialisation yet)
- Overloading print(), making it take in a `StaticArray<char, size>`

```cpp
template<typename T, int size>
class StaticArray {
//....
}

// This only works for StaticArray of size 14
template<>
void print(const StaticArray<char, 14>& array) {
}


// This works for any size
template<int size>
void print(const StaticArray<char, size>& array) {
//...
}
```
- The first one will only work on StaticArrays of size 14, while the second will work for any length.
- The second print() is treated as an **overload** and not a specialisation
	- Why? 
	- Because **partial specialisation not allowed for function templates**
	- Thus it is taken as a completely separate function template

More examples of overloading:
```cpp
template <typename T>  
void print(T x) {  
std::cout << "generic\n";  
}  

// This is a separate overload. NOT partial specialisation
template <typename T>  
void print(T* x) {  
std::cout << "pointer template\n";  
}
```

```cpp
template <typename T>
void print(const T& t) {
    std::cout << t << '\n';
}

void print(double d) {
    std::cout << std::scientific << d << '\n';
}
```
# Ordering - overloadings vs specialisation
- **Overloading is resolved BEFORE template specialisation**
### Step 1: Overload resolution
Build the candidate set. This includes:
- Non-template functions
- **Primary** function templates (after deduction succeeds)

Explicit specializations are **invisible** at this stage.

Ranking among candidates:
1. If a non-template function and a template produce equally good matches, the non-template wins.
2. Among templates, if two primary templates both match, partial ordering determines which is "more specialized" (this is the template partial ordering rule, unrelated to explicit specializations).
3. Among all candidates, standard conversion ranking applies — fewer implicit conversions (like not adding `const`) is preferred.

This step produces exactly one winning candidate.
### Step 2: Template instantiation
Now that a specific primary template has won **and** `T` has been deduced to a concrete type, the compiler asks: "does an explicit specialization of this template exist for **exactly** this `T`?"
- If yes → use the specialization's definition
- If no → instantiate the primary template with the deduced `T`

eg.
```cpp
// Headers

template <class T> 
void foo(T &i) { std::cout << 1; }

template <>
void foo(const int &i) { std::cout << 2; }

int main() {
    int i = 42;
    foo(i);
}
```
- 1 printed
- No overload resolution since there's only one candidate. It only runs on base templates and non-template functions (specialisation not yet)
- T& evaluated against int lvalue -> T = int.
- Is there anything that matches T = int?
	- Only the base. The const specialisation **isn't even valid**

What if overloaded instead of specialisation?
```cpp
template <class T>
void foo(T& i) { std::cout << 1; }

void foo(const int& i) { std::cout << 2; }  // non-template overload
```
- In this case, in step 1, the template is chosen since `int&` is a better match than `const int&` (**less cv-qualification is preferred**)

### Note: ordering matters! Specialisations must come after base
```cpp
template <class T>
void foo(T) { std::cout << 1; }

// Specialisation of the above class.
template <>
void foo<>(int*) { std::cout << 2; }

template <class T>
void foo(T*) { std::cout << 3; }

int main() {
    int *p = nullptr;
    foo(p);
}
```
- `foo<>(int*>)` is a specialisation of foo(T) and not foo(T*), because it's not even aware of foo(T*) yet (declaration order)
- Hence it's a specialisation of foo(T)

- **Overload resolution** first
	- foo(T) vs foo(T*) -> the latter is picked.
	- Thus 3 is printed

# Function template specialisation
- When instantiating function template for given type
	- Compiler stencils out copy of templated function, replaces the template type parameters with the actual types used in variable declaration

- If we want a **different** behaviour for a specific type
	- We can **define non-template function** (overloading)
	- See the print() example above
- Compiler will prefer the pre-defined function, rather than instantiating a version from the template

- We can also use **function template specialisation** to achieve the same effect
```cpp
// Here's our primary template (must come first)
template <typename T>
void print(const T& t){
    std::cout << t << '\n';
}

// A full specialization of primary template print<T> for type double
// Full specializations are not implicitly inline, so make this inline if put in header file
template<> // template parameter declaration with no template parameters
void print<double>(const double& d) { // specialized for type double
    std::cout << std::scientific << d << '\n';
}
```
- Warning: full specialisations are not implicitly inline. If put in header file, should be marked as **inline** to not cause ODR violationss

## Partial template specialisation?? Not for functions.
- However, the above example, print() was not a member function.

```cpp
// now a member function
template<typename T, int size>
void StaticArray<T, size>::print() const {..} 

// ❌ CANNOT do this!
template<int size> // ❌ NO!!!!
void StaticArray<double, size>::print() const {..}
```
- **Cannot partially specialise a function**
	- Alternative 1: partially specialise the whole class, then write an overloaded function
```cpp
template <typename T, int size>
class StaticArray {
	void print() const {..}
}

// partially specialised class
template <int size>
class StaticArray <double, size> {
	void print() const; // defined below
}

template <int size>
void StaticArray<double, size>:: print() const {
//..
}
```
- StaticArray<double, size>::print() is no longer a partially specialised function, it is a non-specialised member of partially specialised class StaticArray<double, size>
	- But, a lot of code reused!

- Alternative 2: use inheritance, with a common base class
```cpp
template <typename T, int size>
class StaticArray_Base {
}

template <typename T, int size>
class StaticArray: public StaticArray_Base<T, size> {
};

template <int size>
class StaticArray<double, size>: public StaticArray_Base<double, size>
```
