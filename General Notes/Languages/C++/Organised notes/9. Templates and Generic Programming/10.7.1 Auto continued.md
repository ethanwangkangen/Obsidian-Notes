# Uses of auto
- Representing types only known compilers, eg.
- lambdas
```cpp
auto lambda1 = [](&something) {return something;};
```
- C++14 : parameters to lambda
```cpp
auto lambda2 = [](const auto& p1, const auto& p2) {return *p1 < *p2};

// type of auto inferred by the methods called on it
```

# Other benefits
- Avoiding uninitialised variables
- Avoiding verbose variable declarations
- Type shortcuts
	- `unsigned sz = v.size()` // but unsigned int may be diff size on diff machines
		- `auto sz = v.size();` // sz's type is `std::vector<int>::size_type
	- `for (const std::pair<std::string, int>& p : m)`
		- where m is `unordered_map<std::string, int>`
		- Problem! The correct type of p is `const pair<const string, int>`. So the compiler will try to convert to the p we have declared, which will result in temporary object created and destroyed at the end, instead of a reference to the element in m.

# Differentiating from std::function
- std::function can also be used to declare a variable that holds a closure
	- BUT not the same!
- std::function uses more memory usually
	- May allocate heap memory to store closure
- std::function slower

# When not to use auto
- Undesirable type deduction
- eg. when `vector<bool>` is involved
```cpp
std::vector<bool> features(..);
auto highPriority = features(w)[5];
// type of auto is no longer bool! undefined bheviour
```
- `auto` will deduce as `std::vector<bool>::reference` (not a real reference, a proxy class implementation), and does not have the corect value of bit 5
- `vector<bool>` is a **proxy class**
	- Class that exists to **emulate and augment** behaviour of some other type
	- Invisible proxy classes should be shileded from clients

## Resolution - explicitly typed initialiser idiom
- Force `auto` to deduce the type you want
```cpp
auto highPriorirty = static_cast<bool>(features(w)[5]);
```

# Trailing return type
```cpp
template <typename T>
decltype(auto) f (T thing) {
	return T;
}

template <typename T>
auto f (T thing) -> decltype(T) {
	return T:
} 
```
- The above examples are the same.
- Trailing return type is usually not strictly necessary, except for examples below

## Trailing return type necessary
- SFINAE 
	- Substitution Failure Is Not An Error
	- When the compiler is trying to figure out which **overload** of a **template** function to use
		- if substituting the template arguments into one candidate produces an **invalid** type or expression, the compiler doesn't error out
			- it just quietly discards that candidate and moves on to the next one.
	- However, SFINAE only operates on **signature, return type and template parameters**
		- **NOT the body!!!!!!!!!!!**
	- Example below
		- `getSize(int)` falls back to second overload in the first scenario
		- However, if `decltype(auto)` is used for `getSize()`then there's nothing invalid in the signature itself, so overload 1 accepted as ccandidate
			- Then `error`
```cpp
// Overload 1: only works if T has a .size() method
template <typename T>
auto getSize(T& container) -> decltype(container.size()) {
    return container.size();
}

// With decltype(auto) — NO SFINAE protection
template <typename T>
decltype(auto) getSize(T& container) {
    return container.size();  // hard error if T has no .size()
}

// Overload 2: fallback for anything
template <typename T>
int getSize(T& thing) {
    return 0;
}
```
