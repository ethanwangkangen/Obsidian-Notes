# This pointer
-  **const pointer that holds address** of current implicit object
	- `simple.setId(2);` -> `Simple::setId(&simple, 2);
	- `void setID(int id) { m_id = id; }` -> `static void setID(Simple* const this, int id) { this->m_id = id; }`
	
- Can return `*this`
	- Primary reason is to allow member functions to be "chained" together
	- This is a dereferencing of the pointer to return the **actual object**
		- So can pass into a **reference** for example (see below)

```cpp
Calc& add(int value) { m_value += value; return *this; }
Calc& sub(int value) { m_value -= value; return *this; }
Calc& mult(int value) { m_value *= value; return *this; }   


// Then can do
calc.add(5).sub(3).mult(4);
```

- Resetting class
```cpp
void reset() {
	*this = {};
}
```

- Checking for object identity (self check
	- Avoiding self-assignment in operators
```cpp
Obj& operator= (const Obj& other) { 
	if (*this == other) return *this; // return reference
}
```

# Header files
- Can define member functions outside of class definitions
	- But remember the class scope, `Obj::func() {..}`
- Unlike functions, compiler needs to see the **full definition** of a class for the type to be used. 
	- Not just a forward declaration
	- Thus, header files usually contain **full definition** of a class rather than forward declaration of class
		- Data members, **inline/constexpr** member functions.
		- Non-inline functions only need declaration in header, definition can be in .cpp

## Common setup
- **Classes** defined in **header** file of same name
- **Member functions** just as declarations
	- defined in .cpp file of the same name
	- those defined **inside** class definition are **implicitly inline**
	- those defined **outside** class definition are not implicitly inline -> hence in a .cpp file
- `#include` the header in the files that you want to use the class in

Why not define member functions inside class definition in header? Since they become inline anyway?
- Compile time explosion
- Binary size bloat
- Encapsulation

### Default arguments
- Put **default arguments** for member functions **inside the class definition (header)**
- Remember that default arguments are **bound at compile time**
	- Substituted where the function is **called**, not defined
- If put in the .cpp,
	- Header only sees the forward declaration
	- Does not see the implementation, cannot substitute the value

eg.
```cpp
//A.h
class A {
	void foo(int x);
};

// A.cpp
void A::foo(int x = 10){}

// main.cpp
#include "A.h"
int main() {
	A a;
	a.foo(); //error
}
```
- When compiling `main.cpp`, the compiler sees only `void foo int();`, so it thinks foo needs an argument but has no idea about the `=10` since that information lives in `A.cpp` which is not seen during compilation of `main.cpp`
	- So there's no substitutino done.

- Default arguments belong to the **caller's compilation step**
	- So defaults must live in headers

> Default arguments : put in headers and not .cpp files


# Nested types
```cpp
class Fruit {
public:
	enum Type {
		apple,
		banana,
		cherry
	};
	//etc
```
- Unscoped enumerators
	- Class types act as **scope region** for names declared within
		- We use unscoped enumerator for Type, so the class becomes its scope
		- `Fruit::Type`, `Fruit::Apple` here.

- Nested classes
	- Do **not** have access to the `this` pointer of outer class
	- However, can access **private members** of the outer class that are in scope

- Forward declarations
	- Nested type can be forward declared within the class that encloses it
		- However can't forward declare nested type prior to outer class definition

```cpp
class outer {
public:
    class inner1;   // okay: forward declaration inside the enclosing class 
    class inner1{}; // okay: definition of forward declared type inside the enclosing class
    class inner2;   // okay: forward declaration inside the enclosing class 
};

class inner2 // okay: definition of forward declared type outside the enclosing class
{
};
```

# Static member variables
```cpp
struct Something {
    static int s_value; // declare s_value as static (initializer moved below)
};

int Something::s_value{ 1 }; // define and initialize s_value to 1 
```
- Not associated with **objects/instances**
- Static lifetime
	- Created at start of program, destroyed at end
- Can call without instance
	- `Class::variable`

## Initialisating static member variables
- Declaration only -> **no memory allocated!**
- Where to **define**?
	- Non-template class
		- `.cpp` file, or
		- **inline/constexpr/const (special case)** and below class definition in header (to prevent ODR)
			- C++17, **constexpr** variables are **implicitly inline**
			- For static members, const integral members get special compile-time exception rule
	- Template class
		- directly underneath template class defintion (no ODR since implicitly inline)
> Make static members **inline** or **constexpr** so they can be initialised inside class defintion

- Only static members can use type deductino (auto, CTAD)

```cpp
class Foo {
    auto m_x { 5 };           // NO! auto not allowed for non-static members
    std::pair m_v { 1, 2.3 }; // NO! CTAD not allowed for non-static members

    static inline auto s_x { 5 }; // auto allowed for static members
    static inline std::pair s_v { 1, 2.3 }; // CTAD allowed for static members
```

# Static member functions
- Have no `this` pointer
- Can be defined outside class definition
	- If defined **outside** class definition, no longer **implicitly inline**. Must make **inline** to not violate ODR, especially if in header file.
- C++ does not support static constructors
	- If static variable can be directly initialised, just initialise it at point of definition.
	- If static variable requires executing some code, there are other ways to do it eg.
		- Use static function, create an object, fill it with data then return to caller.
