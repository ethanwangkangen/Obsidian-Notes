# Constructor
- Note: constructor doesn't create the object!
	- Compiler sets up memory allocation for object prior to constructor call
	- Constructor is then called on the uninitialised object
- Syntax
	- Same name as class
	- For template classes, **exclue template parameters**
	- No return type
	- Non-const. Constructors, unlike normal functions, can be invoked on a const object
- Behaviour
	- **Implicit conversion of arguments**
		- Same as regular functions

## Default constructor
- Constructor with **no arguments**
- If has one,
	- Value initialisation and default initialisation will call the default constructor
	- `Object A;` // default initialisation
	- `Object A{};` // value initialisation
- If **no user defined constructors**
	- **Implicit** default constructor is generated
		- `Foo() {};`
	- Does not initialise any member
	- Can do this explicitly with `Foo() = default;`
		- Pre C++20, explicit default
	- Value initialisation and default initialisation call the implicit default constructor
		- **But ONLY value initialisation {} will zero-initialise the object first**

Summary:
```
T obj;
→ “Call constructor, don’t touch memory beforehand”

T obj{};
→ “Make it zero first — unless the class defines its own constructor”
```


## Zero initialisation
Happens when:
- Object has **static storage duration** (eg. global values, static values)
- **Value** initialisation of class without user-provided constructor (see above)
	- `Object A{};`
- **Value** initialisation of built-ins or arrays
	- `int x{};`
- As part of initialising **subobjects** during **value** initialisation
## Using constructors
```cpp
Foo{2,5} // direct list initialisation. prevents narrowing conversion
Foo(2,5) // watch out!

// If class has constructor that takes std::initializer_list, {} prefers.
vector<int> v {1,2,3,4,5}; // calls initializer_list constructor
vector<int<> v2 (3,5) // calls constructor for 3 elements, all of value 5
```

- Most vexing parse
	- `Type name (something()));`
	- `Type name ();`
```cpp
Foo a{}; // ok.
Foo a(); // Most vexing parse - will get mistaken for function declaration!
```

## Member initialiser list
```cpp
Foo (int x, int y) : m_x {x}, m_y {y} { // m_x and m_y initialised
	//other stuff
}
```
- **More efficient** than initialising in the constructor, since constructor is actually **assignment**
- Also necessary for **const members**
- Priorities
	- If member is listed in the **member initialiser list**, then that initialisation value is used
	- Otherwise, if member has **default member initialiser**, then that is used
	- Otherwise, member is **default-initialised** (for fundamental types this is left **uninitialised**)

## Copy initialisation
- Calls the **copy constructor** (implicit one, if not defined)
	- Function signature is `Foo (const& Foo)`
	- See [[5.3 Copying]]
- **Implicit conversions** use this
	- Temporary object on RHS constructed, then **copied** into target object on the left.
- Explicit constructors not allowed
- Conceptually, temporary object created on RHS, then copy constructor called to initialise the LHS. However, due to **eliding**, in case like the first one below, copy constructor is not called.
- `Foo f3 = 3;` 
- `Foo f3 = {6};` 
- `Foo f7 = f3;` // Uses Foo(const Foo&), ie copy constructor
- `Foo f7 = {f3};`

## Delegating constructors
- **Overloaded** constructors, want to call one inside another - how?
- **Don't** call constructors within **the body** of another function
- Instead, **call the constructor in the member initialiser list**
	- The constructor with fewer parameters should delegate/call the one with more parameters
	- Note: constructor that delegates to another **cannot** do initialisation
```cpp
Foo(int x) : Foo {x, 0} {} // this delegates to the bottom one
Foo(int x, int id) : m_x {x}, m_id {id} {}
```

## Converting constructors
- Constructor thatcan be used to perform an **implicit conversion**
- By default, **all** constructors are converting constructors

## Explicit constructors
- `explicit Foo() {}`
- Prevents compiler from using this constructor for implicit conversions
- **Prevents** 
	- **copy** intialisation/**copy list** intiaisation
	- **implicit conversions**
		- Since this uses copy initialisation/cop list initialisation
	- implicit conversions from **return by value**
- Still can be used for direct and direct list initialisation

> Best practice: make any **constructor** that takes in a **single argument** explicit by default
> Do not make copy/move constructors explicit since these don't perform conversions

- **KIV: explicit on conversion operators and multi-argument cnstructors**
```cpp
// prevent implicit conversion construction
struct Radius {
    explicit Radius(double r) { ... }
};

drawCircle(5.0);          // ❌ error — no implicit conversion
drawCircle(Radius(5.0));  // ✅ fine — explicit conversion

// prevent implicit conversion from return by value
class Foo {
	explicit Foo(int x) {
	}
};

Foo getFoo() {
	return 5; // fail, can't implicitly convert int to Foo
	return {5}; // fail, can't implicitly convert initialiser list to Foo
	return Foo{5}; //ok
}

```

# Inheriting constructors
 - `using Base::Base;` 

This is a **using-declaration** (C++11+) that **brings base-class constructors into the derived class**.

```cpp
template<typename T>
class Vec : public std::vector<T> {
public:
    using std::vector<T>::vector;  // inherit constructors
};
```

Not avised when
- Derived adds new members
- Derived has invariants to enforce
- Multiple inheritance
# Initialisation summary
```
Initialisation
├─ Default            : T x;              // garbage / default ctor
├─ Value              : T x{};            // zero / default ctor
├─ Direct             : T x(y);           // ctor / conversion, narrowing allowed
├─ Copy               : T x = y;          // ctor / conversion, implicit
├─ List (Brace)       : T x{y};           // ctor / conversion, narrowing prohibited
├─ Aggregate          : T x{a,b,c};       // member-wise, zero for missing
├─ Reference          : T& r = obj;       // must bind to object
└─ Temporary object   : T(y) / T{y}       // lifetime till end of expression
```

Summary
```cpp
struct Point { int x; int y; };
class MyClass {
public:
    MyClass() { /* default ctor */ }
    MyClass(int a, int b) { /* param ctor */ }
};

int main() {
    int a;        // default-initialized -> garbage
    int b{};      // value-initialized -> 0

    Point p1;     // default-initialized -> x,y garbage
    Point p2{};   // value-initialized -> x=0, y=0
    Point p3{1,2};// direct list-init -> x=1, y=2

    MyClass m1;   // default-initialized -> default ctor
    MyClass m2{}; // value-initialized -> default ctor
    MyClass m3(1,2); // direct-initialized -> param ctor
    MyClass m4{1,2}; // direct list-init -> param ctor
    
    vector c{1,2,3,4,5}; // std::intiialiser_list constructor
}
```

# Temporary objects
- Can create temporary objects (and pass them to function) in 2 ways
	- `fn(IntPair{5,6});` // Default initialisation
	- `fn({5,6});` //Compiler **deduces necessary type from parameter**
```cpp
// Case 1: Pass variable
IntPair p { 3, 4 };
print(p);

// Case 2: Construct temporary IntPair and pass to function
print(IntPair { 5, 6 } );

// Case 3: Implicitly convert { 7, 8 } to a temporary Intpair and pass to function
print( { 7, 8 } );
```

## Casting: static_cast vs explicit instantiation of temporary object
```cpp
std::cout << static_cast<int>( c ) << '\n'; // static_cast returns a temporary int direct-initialized with value of c
    std::cout << int { c } << '\n';             // explicitly creates a temporary int list-initialized with value c
```
- Static cast
	- Internal syntax: `int tmp (c)`
	- Temporary object **direct-initialised**
- Explicity cast
	- Internal syntax: `int temp {c}`
	- Temporary object **list-initialised**
		- **NO narrowing allowed**

> Prefer static_cast when converting to a fundamental type, and a list-initialized temporary when converting to a class type.

# Destructors
- Called **automatically** when object of non-aggregate class type is destroyed
- Only **one** allowed
- Object only destroyed **after** destructor executes
	- Can call member functions and do other cleanup stuff inside
- Implicit destructor
	- If non-aggregate class type has no user-declared destructor, compiler generates destructor with empty body

> Generally, don't call destructor explicitly

- **Unhandled exceptions** will cause program to terminate, and may not unwind stack before doing so.
	- If stack unwinding does not happen, **destructors will not be called**

```cpp
~Object() {
	// something
}
```
