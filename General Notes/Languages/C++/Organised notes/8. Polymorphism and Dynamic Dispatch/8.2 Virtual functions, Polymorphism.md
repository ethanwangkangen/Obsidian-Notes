# Virtual Functions
- Resolves to **most-derived** version of the function for the actual type of the object being referenced/pointed to
	- Only works if called through **pointer/reference**

```cpp
Base {
public:
    virtual int getValue() const  { return 5; }
};

class Derived: public Base {
public:
    virtual int getValue() const override { return 6; }
};
```

> Never call virtual functions from constructors/destructors, since Derived portion is not created yet (for constructor), or destroyed first (for destructor)

- If a function is marked as virtual, **all matching overrides** in derived class are **implicitly virtual**, even if not explicitly marked
## Override keyword
- See example above
- Doesn't do anything functionally.
	- Just for compiler to **check** that function is really over riding
- Enforces this rule
	- This function must **match a virtual function in a base class exactly**
		- Same name, param types, const/ref qualifiers, correct inheritance chain
## Final keyword
- Restrict from overriding further
```cpp
class A{
public:
	virtual std::string_view getName() const { return "A"; }
};

class B : public A {
public:
	// note use of final specifier on following line -- that makes this function not able to be overridden in derived classes
	std::string_view getName() const override final { return "B"; } // okay, overrides A::getName()
};
```
- `getName()` can't be over-ridden for classes that inherit from B

# Virtual Destructors
- We want the object that is pointed to by the pointer/reference to have its correct inherited destructor called.
- **Always make destructor virtual when class is inherited!!!!**

> Whenever you are dealing with inheritance, you should make any explicit destructors virtual.

```cpp
virtual ~Base() // note: virtual
    {
        std::cout << "Calling ~Base()\n";
    }
```

> If you intend your class to be inherited from, make sure your destructor is virtual and public.
>  If you do not intend your class to be inherited from, mark your class as final. This will prevent other classes from inheriting from it in the first place, without imposing any other use restrictions on the class itself.
# Calling the base version instead
- Use **scope resolution** -> eg. `Base::function()`

# Binding and dispatching
- Binding: associating names with properties
- Function binding: process that determines what function definition is associated with a fcuntion call

## Early binding
- Direct call made to non-member function / non-virtual member function
	- Resolved at compile time
```c
// assembly code
mov     edi, 5           ; copy argument 5 into edi register in preparation for function call
call    printValue(int)  ; directly call printValue(int)
```
- Overloaded functions can also be resolved via early binding

## Later binding
- Function call can't be resolved until runtime, ie. not during compile time.
- Examples
	- Function pointer
```cpp
auto fcn { printValue }; // create a function pointer and make it point to function printValue
fcn(5);        

// corresponding assembly code
lea     rax, [rip + printValue(int)] ; determine address of printValue and place into rax register
mov     qword ptr [rbp - 8], rax     ; move value in rax register into memory associated with variable fcn

mov     edi, 5                       ; copy argument 5 into edi register in preparation for function call
call    qword ptr [rbp - 8]          ; invoke the function at the address held by variable fcn         
```

- Vtables
	- Dynamic dispatch
	- Every class that uses virtual functions (or is derived from class that uses virtual functions) has a corresponding vtable
		- **NOT each object!** One vtable per class. Then the object has a pointer that points to that table.
	- One entry for each virtual function that can be called by objects of the class
		- Contains function pointer that points to the **most-derived function** accessible by the class
	- Compiler adds a hidden pointer that is member of bass class, which is `*__vptr`, ie pointer to the virtual table
		- Inherited by classes that inherit it
		- But the value it holds is different (since it points to a different Vtable)

## Pure virtual functions
```cpp
virtual int getValue() const = 0; // a pure virtual function
```
- Any class with at least one virtual functions becomes **abstract base class**
	- Cannot be instantiated
- Any class with pure virtual function **should also have virtual destructor**

- Abstract classes still have virtual tables
	- Generally contain null pointer, or point to generic function that prints an error
### Interface class
- Class that has **no member variables** and **all functions are pure virtual**
- Similar to interface classes in Java
- **No constructor**
- **Virtual destructor**

```cpp
class Container {
public:
	virtual double& operator[](int) = 0;
	virtual int size() const = 0;
	virtual ~Container() {}
}
```
- Concrete class that inherits from Container has to implement the virtual functions
- Functions can then specify Container& / Container* as parameters, and the concrete class can be passed in
- **MUST** be passed through **reference/pointer**
	- **Cannot** pass by **value**, since abstract classes are **non-instantiable**
```cpp
class ListContainer : public Container {
//
}
ListContainer a;
void foo(&Container);
foo(a);
```

- Usually, should **delete** the default move and copy operations for such classes
	- Because, given a pointer to the base, we don't know what members the derived class has, so we don't know how to copy them
```cpp
Shape(const Shape&) = delete; // copy constructor
Shape& operator=(const Shape&) = delete; // copy assignment

Shape(Shape&&) = delete; // move constructor
Shape& operator=(Shape&&) = delete; // move assignment
```
## Virtual base class
- Diamond inheritance problem
	- 2 instances of the first base class constructed!
![[Pasted image 20260413152816.png]]

- Virtual base class
	- Only one base object
	- Shared between all objects in the inheritance tree and is only constructed once
	
```cpp
class PoweredDevice {
};

class Scanner: virtual public PoweredDevice {
};

class Printer: virtual public PoweredDevice {
};

class Copier: public Scanner, public Printer {
};
```
- Scanner, Printer, **AND COPIER** all instantiate a PoweredDevice in their constructor member initialiser list
	- BUT **only copier** actually constructs the PoweredDevice, (when copier is constructed)
	- If we instantiate Printer directly, then of course Printer constructs the PoweredDevice
	- Also this is the one time when Copier is allowed to call a non-immediate-parent constructor directly

- For constructor of most derived class, virtual base classes always created before non-virtual base classes
	- Ensures all bases get created before derived classes

- If class inherits one or more classes that have virtual parents, the **most derived class** is the one responsibel for construcing the virtual base class (even in singae inheritance case)

- All classes inhering virtual base class will have a virtual table, even if they normally would not have one otherwise
