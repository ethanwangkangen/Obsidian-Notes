# Throw
- Can throw any data type
```cpp
throw -1;
throw ENUM_INDX;
throw "STRING";
throw dX;
throw MyException("Error"); // throw an object of class MyException
```

# Looking for, and handling exceptions
- Try and catch
```cpp
try {
	...
	throw -1;
} catch (int x) {
	...
}
```
- Try block detects any exceptions that are thrown by statements within the try block,
	- routes them to a catch block with a matching type for handling
- Try blocks catch exceptions not only from statements within the try block, but also from **functions that are called within the try block**.

# Exception handling
- When exception is raised using **throw**
	- Running program finds the **nearest enclosing try block** (propagating up the stack if necessary to find an enclosing try block) to see if any of the **catch handlers** attached to the try block can handle that type of exception
	- If function can handle the exception, it does so
- If function cannot handle,
	- Function caller (nextt function up the call stack) can handle the exception
		- The call to the current function must be inside try block, and matching catch block associated
		- And so on recursively
- If **matching exception handler found**
	- Execution jumps to the top of the matching catch block
	- This requires **unwinding the stack**
		- Variables are destroyed; have their **destructors called
			- Stack variables destructors called **automatically** when scope ends
			- For heap variables, not so simple. Destructor needs to be called by `delete`, so this may not happen (hence the need for RAII and smart pointers)

# Uncaught exceptions
- If exception unhandled, **call stack may or may not be unwound**
	- Local variables not destroyed

# Function try blocks
- When need a constructor to handle the exception thrown in member initialiser list
```cpp
class B : public A {
public:
	B(int x) try : A{x} // note addition of try keyword here
	{
	}
	catch (...) // note this is at same level of indentation as the function itself
	{
                // Exceptions from member initializer list or
                // from constructor body are caught here

                std::cerr << "Exception caught\n";

                throw; // rethrow the existing exception
	}
};
```

- Howeve, no return statements allowed, and not allowed to resolve exceptions (for constructor)

| Function type                | Can resolve exceptions  <br>via return statement | Behavior at end of catch block |
| ---------------------------- | ------------------------------------------------ | ------------------------------ |
| Constructor                  | No, must throw or rethrow                        | Implicit rethrow               |
| Destructor                   | Yes                                              | Implicit rethrow               |
| Non-value returning function | Yes                                              | Resolve exception              |
| Value-returning function     | Yes                                              | Undefined behavior             |

# Exceptions and destructors
- Exceptions should **never** be thrown in destructors
	- Unlike constructors, where throwing exceptions can be useful way to indicate that object create did not succeed
	- Problem: exception thrown out of destructor during stack unsinding process -> continue undwinding or not? Compiler doesn't know -> terminates
