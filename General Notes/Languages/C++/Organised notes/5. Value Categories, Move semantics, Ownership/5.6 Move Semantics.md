- Instead of deep copying the pointer, **just move ownership of the pointer**
# Move Assignment

```cpp
Obj operator=(Obj&& a) noexcept {
	if (&a == this) return *this;
	
	// Release any resource we holding
	delete m_ptr;
	
	// Transfer ownership of a.m_ptr to m_ptr
	m_ptr = a.m_ptr;
	a.m_ptr = nullptr;
	return *this;
}
```

# Move constructor
```cpp
Obj (Obj&&a): noexcept
	m_ptr : {a.m_ptr} {
	a.m_ptr = nullptr;
}
```

# Disabling moving
- Delete copy constructor
	- Note that if we do so, compiler will **not** generate implicit move constructor
```cpp
Obj& operator=(Obj&& a) = delete
```

# Copy vs move function signature summary
```cpp
copyFunction(const T& value); // copy. accepts l-value, both const and non const, and r-values also (temp created)
moveFunction(T&& value); // accepts r-values only
```
- Note that while `const T&` can bind to r-value, the latter is preferred during resolution, since C++ overload resolution uses reference binding rules
	- Exact match beats binding to const
	- T&& matches r value exactly
	- const T& requires binding a const reerence to an rvalue, less exact

# std::move
- Moves l-value object into an r-value reference, so that move assignment/constructors can use it

```cpp
std::string str = "hello";

std::move(str);
// is basically
static_cast<std::string&&>(str);

// allowing things like
v.push_back(std::move(str));
// overloaded function that takes in r-value
```

- eg. 
```cpp
template <typename T>
void mySwapCopy(T& a, T& b) {
	T tmp { a }; // invokes copy constructor
	a = b; // invokes copy assignment
	b = tmp; // invokes copy assignment
}

template <typename T>
void mySwapMove(T& a, T& b) {
	T tmp { std::move(a) }; // invokes move constructor
	a = std::move(b); // invokes move assignment
	b = std::move(tmp); // invokes move assignment
}
```

- **Moved-from objects** will be in a valid but possibly **indeterminate state**
	- It is safe to call any function that **does not depend on the current value of the object**
		- Can **set, reset value, operator=, clear(), reset() etc.**

# Returning from a function returns rvalue, no need to std::move()
- Compiler treats **local variables returned from a function** as rvalue, so no need to call std::move() on it
	- In fact doing os can hurt by preventing NRVO

# What happens when we try to move a const object?
- eg.
```cpp
const std::string msg = "hello";
process(std::move(msg));
```
- The result of `std::move(msg)` is a **const std::string&&**
	- `process(std::string&&)`  CANNOT bind to this!!
	- Hence `process(const std::string&)` called
- Essentially
	- Move function **not called**
	- **COPY function called**

# Move constructors should be noexcept
- Why?
	- When STD containers reallocate, they will either use the **move** constructor or **copy** constructor, depending on whether or not the move constructor is noexcept
	- If and only if move constructor is **noexcept**, it will use it
		- Rationale: If we **throw** during a `push_back()`, we want to leave the container **unchanged**
		- What happens if we throw during move? Don't know how to go back to original container anymore.
