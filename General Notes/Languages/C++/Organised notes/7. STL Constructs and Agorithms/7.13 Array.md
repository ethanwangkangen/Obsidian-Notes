# Array decay 
```cpp
static int arr1[] {1, 2}; // int[2]
static int arr2[] {1, 2, 3}; // int[3]

struct Struct {
    int (&a)[2] = arr1; // int (&) [2] 
    int b[2] {1, 2};  // int[2]
    int c[3] {1, 2, 3}; // int[3]
};

int main() {
    auto& x = arr1; // int (&) [2] - reference to int[2]
    auto y = arr1; // int * - because of array decay
    auto z = arr2; // int * - because of array decay

    Struct s {};
    s.a; // int (&) [2]
    s.b; // int [2]
    s.c; // int [3]

    std::cout << (typeid(arr1) == typeid(x)); // 1
    std::cout << (typeid(arr1) == typeid(y)); // 0
    std::cout << (typeid(arr1) == typeid(s.a)); // 1
    std::cout << (typeid(arr1) == typeid(s.b)); // 1
    std::cout << (typeid(arr1) == typeid(s.c)); // 0

    std::cout << (typeid(z) == typeid(y)); // 1
    std::cout << (typeid(arr2) == typeid(s.c)); //1
}
```

# When does array decay happen
> Decay happens whe you use an array where a VALUE is EXPECTED
- Assignment/initialisation by value
	- `auto y = arr`
	- `int* p = arr;`
	- Copying, and can't copy an array, so it decays to a pointer
- Function argument pass by value
	- `void f(int a[])` is actually `void f(int* a)`
- Most expressions/operators
	- Using array in arithmetic (`arr+1`). comaprisons, ternary etc.

### When it DOESN'T decay
- `&arr` - address-of gives you pointer-to-array, not pointer-to-pointer
- `sizeof(arr)` gives full array size, not poiner size
- `typeid(arr)` inspects declared type
- `decltype(arr)` same, inspects declared type
- Binding to a reference - `auto& x = arr` or `int (&r)[2] = arr`
