# extern
2 use cases
- 1. With initialisation -> to give external linkage to const global variables
- 2. Without initialisation -> to declare extern forward declarations, ie. this variable is defined in another file

# inline
- Used to be to request **inline expansion**, but this is outdated.
	- Don't do this for performance anymore. May not even be more performant.

## Inline functions
- Allowed to have multiple definitions.
- Constexpr function -> inline
	- Implicitly
	- So that can put in headers safely, and can use in compile time
- Requirements
	- Compiler must see **full definition** in **each translation unit** where function is used (not just forward declaration)
	- Every definition for inline function must be **identical**
- Use case
	- Function within **header file**

## Inline variables
- Same as above, allowed to have multiple definitions. Same requirements.
- Constexpr variable -X-> inline!
	- Must explicitly mark `inline constexpr`
- By default, **external linkage**.

# static
| Context                  | Effect of `static`                                                                                             |
| ------------------------ | -------------------------------------------------------------------------------------------------------------- |
| Global variable/function | Give **internal linkage** to a function/non-const variable                                                     |
| Local variable           | Change lifetime to static rather than automatic (last throughout program). Note that **scope does not change** |
| Class member variable    | Shared across all instances; belongs to class. Eg. `count`                                                     |
| Class member function    | Can be called without an object; cannot access non-static members. eg. `utility_function()`                    |

# Global

## Global Variables
### Const
- **Must be initialised**
- By default, **internal linkage**
	- `constexpr int g_x{1};`
- To give **external** linkage, use `extern` or use `inline`
	- `extern const int g_x{1};`
		- This would be in a .cpp file, then `extern const g_x;` forward declaration inside a .h file which would be `#included` in many files.
		- Downside: forward declarations in those files can't be `constexpr`.
	- `inline constexpr int g_x{1}'`
		- Can use this direclty in .h file.

Why const **internal linkage** by default?
- To use in headers without violating ODR

Why not use `constexpr` with `extern`?
- Doesn't make sense. 
- If another translation unit uses `extern constexpr int x;`, compiler won't see the initialiser, so can't use it as a compile time constant anyway.

### Non-const
- May or may not be initialised.
- **Do not use extern** if unnecessary.
- By default, **external linkage**. Do not use `extern` or C++ mistake for forward declaration (below).
	- `int g_x;`
	- `inline int g_x;`
- To give internal linkage, use `static`
	- `static int g_x;`

> Note: try to avoid non-const global variables.


## Forward declarations
Forward declaration with `extern`: (second use case for `extern`)
- If `extern` is used **without initialisation**, this means "this variable has been defined somewhere else, in another file"
	- `extern int g_x;`

> Note: **cannot** use `constexpr` as forward declaration, since `constexpr` variables need entire definition during compile time


## Global Functions
- By default, **external linkage**
	- `int add() {..}`
- To give internal linkage, use `static`
	- `static int add() {..}`


# Flowchart and cheatsheet

What are you declaring? 
│
├─ Is it a function? ──> Yes
│      ├─ Is it constexpr? 
│      │      └─ Yes → implicitly inline → header safe → usable at compile-time
│      │
│      └─ No → external linkage by default
│             └─ Want TU-only? → use `static` → internal linkage
│
└─ No (it's a variable)
       ├─ Is it constexpr?
       │      ├─ Yes
       │      │     ├─ Want header-only / used in multiple TUs? → use `inline constexpr`
       │      │     └─ Else → define in one .cpp only, cannot forward-declare
       │
       ├─ Is it const? (need runtime initialisation)
       │      ├─ Yes
       │      │     ├─ Default: internal linkage → safe to put in header, each TU gets its own copy
       │      │     └─ Want single shared constant? → declare `extern const` in header, define in one .cpp
       │
       └─ Non-const variable
              ├─ Default: external linkage → avoid if possible
              └─ Want TU-only? → use `static` → internal linkage


# **Key Decision Rules to Remember**

1. **Functions:**
    - `constexpr` → implicitly inline → safe to put in headers.
    - Normal functions → external linkage by default; `static` for TU-only.

2. **Variables:**    
    - `constexpr` → **not inline by default**, must mark `inline` for header-only usage (C++17+).
    - `const` → **internal** linkage by default → safe in header, private to TU. 
	    - Use `extern const` for shared runtime constants.
    - Non-const → default external linkage → avoid if possible; `static` for TU-only.
    
3. **`extern` without initializer** → forward declaration of variable defined elsewhere.

4. **TU-only** → variable or function is visible **only in this translation unit**, safe from linker conflicts. Achieved with `static` or anonymous namespace (for variables/functions).
