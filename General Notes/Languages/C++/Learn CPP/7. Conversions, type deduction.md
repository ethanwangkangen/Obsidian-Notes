# Numeric promotions (for efficiency)
- Narrower numeric types (eg. char) to wider numeric types (eg. int, double) that can be processed more efficiently
- Value preserving
- Also lets functions that take in `int` be called with `char`, `short` etc.
- Types
	- Float -> double
	- Integral promotions
		- signed char/short -> int
		- unsigned char, char8_t, unsigned short -> int (if can hold entire range of type), else unsigned int
		- if char is signed by default, followed the signed char conversion rules above. if unsigned, follows the signed rules above.
		- bool -> int (1 or 0)

**Rule: if type is smaller than int, it gets promoted to int or unsigned int during evaluation**
- Evaluation eg. arithmetic, comparison
- However they are still **stored** as the original type! Promotion is temporary

Types affected
- bool
- char
- signed char
- unsigned char
- short
- unsigned short

eg.
```CPP
printInt(int a) { // this can take in char
}

char a{10}; //char -> int
char b{20};
auto c = a + b; //during arithmetic, compiler promotes a and b to int first
std::cout<<c; //c is actually an int.
```

# Numeric conversions
5 types
- integral to other integral (eg. int -> short, int -> long, short -> char)
- floating point to other floating point (double, loat, long double)
- floating to integral
- integral to floating
- integral/floating to bool

## Casting (use if narrowing)
```CPP
static_cast<int> d
```
- Brace initialisation disallows narrowing conversions
	- Need static cast

## Constexpr clause
- Some conversions are no longer narrowing if the original value (being converted) is constexpr, and value can be stored exactly in destination type
	- eg.
```CPP
constexpr int n1{4};
unsigned int u1{n1}; //narrowing but allowed due to constexpr exclusion
```
- Warning: don't use the above for constextr floating point type to a narrower floating point type (may result in loss of precision still)
- Remember that constexpr things include
	- constexpr variables
	- constexpr literals, eg. `5`
	- return values of constexpr functions
	- constexpr constructors/objects
## Example

```cpp
int main()
{
    int a { 5 }; // no conversion
    int b { 'a' }; // promotion, char->int
    int c { 5.4 }; // narrowing, not allowed
    int d { true }; // promotion, bool->int
    int e { static_cast<int>(5.4) }; // explicit narrowing cast, double->int

    double f { 5.0f }; // float->double
    double g { 5 }; // int->double

    // Extra credit section
    long h { 5 }; // int->long

    float i { f }; // narrowing, double->float not allowed.
    float j { 5.0 }; // **5.0 is a constexpr!** and fits in float, so ok. doube -> float implicit narrowing conversion allowed. see constexpr clause above

}
```
# Arithmetic conversions
- eg. `auto {2+3.5}`

Rank:
- long double
- double
- float
- long long
- long
- int

Rules
   1.
- If 1 operand is integral and other floating, integral -> floating type
- Else, integral operands numerically promoted (see above)

2. 
- After promotion, if 1 operand signed and 1 unsigned, special rules below
- Else, operand with lower rank converted to type with higher rank

## Signed and Unsigned
- If the rank of the unsigned operand is greater than or equal to the rank of the signed operand, the signed operand is converted to the type of the unsigned operand.
- If the type of the signed operand can represent all the values of the type of the unsigned operand, the type of the unsigned operand is converted to the type of the signed operand.
- Otherwise both operands are converted to the corresponding unsigned type of the signed operand.

# Casting types
- static_cast
	- `static_cast<int>(3.0)`
	- int->double, char->int etc.
- dynamic_cast (will revisit)
	- Safe runtime check for polymorphic types (classes with virtual functions)
	- Only works with pointers/references to classes with at least one virtual function
	- Can check if base pointer actually pointer to derived objects
	- Returns nullptr for pointer casts if cast fails, throws `std::bad_cast` for references
eg.
```CPP
class Base { virtual void f() {} };
class Derived : public Base {};

Base* b = new Derived;
Derived* d = dynamic_cast<Derived*>(b); // safe downcast
```
- const_cast
	- `int* ptr = const_cast<int*>(&x);`
	- add/remove `const` or `volatile` froma variable
	- often used to call a function that wrongly doesn't take a const reference
- reinterpret_cast
	- convert any pointer type to another pointer type
	- `char* p = reinterpret_cast<char*> (&a)`
- C-style casts
	- eg. `double(x)` or `(double) x`
	- Tried to perform const, static, reinterpret in order.
	- One thing can do with this that can't do with C++ style casts: **convert derived object to base class that is inaccessible (eg. cos it was privately inherited)**
> Try to avoid C-style casts.

# Typedef and aliases
- Alias
	- eg `using Distance = double;`
	- Then can use `Distance` in future
		- `Distance miles{1.4};`
	- Follow same scoping rules as variable identifiers
		- If inside block, block scope. If in global namespace, has global scope
		- If need type aliases across multiple files, can be defined in header file and `#included` into any files that need it
- Typedef
	- `typedef double Distance;`
	- Same use as alias

> Prefer alias to typedef


# Type deduction/inference
- `auto`
- eg.
	- `auto d { 5.0 }; // 5.0 is a double literal, so d will be deduced as a double`
- Note: prior to C++17, `{}` initialisation will deduce type to be that osf `std::initializer_list<>` instead.
- Must have something to deduce from, ie. need to copy/direct initialise, not just declare
- `const` dropped
	- To keep, use `const auto`
- String literals
```CPP
auto s { "Hello, world" }; // s will be type const char*, not std::string
auto s1 { "goo"s };  // std::string literal
auto s2 { "moo"sv }; // std::string_view
```
- `constexpr
	- not even part of the type system. 
	- implicitly const, but const dropped (see above)
	- can put it back with `constexpr`

> Use type deduction for variables when type of object doesn't matter.
> Favor explicit type when require a specific type that differs from initialiser, or when object is used in context where seeing the type is useful


## Type deduction for functions
```CPP
auto add(int x, int y)
{
    return x + y;
} // return statement returns int, so function deduces int return
```
- Forward declaration insufficient. Must be fully defined before use
> Try to avoid type deduction for functions

- Cannot use in parameters.
	- Works in C++20, but is not invoking type deduction, instead is using *function templates*
