Eg.
```cpp
#include <iostream>

constexpr double calcCircumference(double radius) // now a constexpr function
{
    constexpr double pi { 3.14159265359 };
    return 2.0 * pi * radius;
}

int main()
{
    constexpr double circumference { calcCircumference(3.0) }; // now compiles

    std::cout << "Our circle has circumference " << circumference << "\n";

    return 0;
}
```
- calcCircumference() is a constexpr function, so can be used in a constant expression below

- **CAN** be evaluated at compile time
	- If not, they will just return a runtime result
- To evaluate at compile time,
	- Call to constexpr function must have arguments that are constant expressions
	- All statements and expressions within the constsexpr functino must be evaluatable at compile time
- Note that compile time evaluation of constexpr functions is only **guaranteed** when a constant expression is **required**
	- If not, compiler is free to decide.
		- eg. `int x { calcCircumference(2.0) }` -> compiler chooses if this is done at compile time
- Also for that reason, if a constexpr function is written such that it actually cannot be resolved at compile, this will only be caught by compiler if tried to use it in a constant expression.
	- eg. foo(5) is a constexpr function that actually can't evaluate at compile time
	- `int x {foo(5)}` //ok
	- `constexpr int y {foo(5)}`//compilation fails
	- Thus, **test constexpr functions by using them in context that requires a constant expression**

## Parameters
- Reminder that parameters cannot be constexpr
	- Recall that if need parameters that are constant expression, use **Non-type template parameters**

## Constexpr function -> Inline
- Implicitly inline
	- Compiler needs to see  full definition of constexpr function, not just forward declaration
	- Thus implicitly inline, to allow definition included into each translation unit

> If need use in multiple files, include in headers
> Else if just in one file, put it in the same file


## Consteval, Forcing constexpr function to evaluate at compile time
- Ensure return value is used where a constant expression is required (see above)
	- eg. use return value to initialise constexpr variable
	- But this is per-call basis
- C++20 - **consteval**
	- `consteval int greater (int x, int y) {..}`
- There is **no** current way to tell whether a function call is evaluating at compile time or runtime
	- `std::is_constant_evaluated` returns whetehr or not compiler is being **forced** to evaluate at compile time
	- `if consteval` is just `if (std::is_constant_evaluated())`


## Other notes
- constexpr/consteval functions can use non-const local variables
	- Compiler will just 'execute' the function and return the calculated value
- constexpr/consteval functions can use **function parameters** and **local variables** as arguments in constexpr function calls
	- Generally, if use **non-const** variables in a constexpr function call, this is just evaluated at runtime.
	- But this is the exception.
	- eg,
```cpp
constexpr int goo(int c) // goo() is now constexpr
{
    return c;
}

constexpr int foo(int b) // b is not a constant expression within foo()
{
    return goo(b);       // if foo() is resolved at compile-time, then `goo(b)` can also be resolved at compile-time
}
```
- constexpr function can call a non-constexpr function, but only when the constexpr function is being evaluated in non-constant context

> Prior to C++23, standard says that constexpr function must return constexpr value for **at least one** set of arguments

# When to make a function constexpr
- If function **can** be evaluated as part of a required constant expression
- Pure function
	- Function always returns same result if same arg given
	- No side effects (doesn't change value of static local or global variables, no input output..)
