- Resolving overloaded operators
	- If **all** of operands are fundamental data types, compiler will call built-in routine if one exists. If none exist, compiler error
	- If **any** of operands are program-defined types (my own class/enum type), compiler uses **function overload resolution algorithm (see previously)** to find best match
		- May involve implicitly converting one/more operands to match parameter types of overloaded operator
		- May also involve implicitly converting program-defined types to fundamental types (via overloaded typecast)

## Limitations
- Operators that **can't** be overloaded
	- `:`,  `?`, `.`, `.*`, `typeid`, casting operators
- Can only overload operators that exist
- At leaste one of the operands in overloaded operator must be user-defined type
	- STL classes are considered **user defined**, so technically can define eg. `operator+(double, std::string)`
	- But not recommended. 
> Overloaded operator should operate on at leaste one program defined type
- Cannot change number of operands an operator supports
- All operators keep default precedence and associativity, this cannot be changed

## Best practices
> Keep function of operator as close to intent of operator
> If meaning of overloaded operator is not clear and intuitive, use named function
> Operators that **do not modify** operands (eg. arithmetic) should generally return by **value**.
> Operators that **modify** their **leftmost** operand (eg. pre-increment, any assignment) should generally return the **left most operand by refernece**

# Overloading of specific functions
## Arithmetic - friend functions
```cpp
class Cents {
private:
	int m_cents {};

public:
	explicit Cents(int cents) : m_cents{ cents } { }

	// add Cents + Cents using a friend function
	friend Cents operator+(const Cents& c1, const Cents& c2);

	int getCents() const { return m_cents; }
};

// note: this function is not a member function!
Cents operator+(const Cents& c1, const Cents& c2) {
	// use the Cents constructor and operator+(int, int)
	// we can access m_cents directly because this is a friend function
	return Cents { c1.m_cents + c2.m_cents };
}
```
- Note: friend functions **not** member of class, but can still be defined inside the class if desired

## Arithmetic - Normal functions
```cpp
Cents operator+(const Cents& c1, const Cents& c2)
{
  // use the Cents constructor and operator+(int, int)
  // we don't need direct access to private members here
  return Cents{ c1.getCents() + c2.getCents() };
}
```
- The only difference: need to provide function prototype in header file so other files knows this function exists
```cpp
//cents.h
Cents operator+(const Cents& c1, const Cents& c2);
```

## Overloading I/O
### <<

```cpp
std::ostream& operator<< (std::ostream& out, const Point& point) {
    // Since operator<< is a friend of the Point class, we can access Point's members directly.
    out << "Point(" << point.m_x << ", " << point.m_y << ", " << point.m_z << ')'; // actual output done here

    return out; // return std::ostream so we can chain calls to operator<<
}
```

### >>
```cpp
std::istream& operator>> (std::istream& in, const Point& point) {
	// Danger: partial extraction
	in >> point.m_x >> point.m_y >> point.m_z;
	
	// This is the better way.
	double x{}; double y{}; double z{};
	if (in >> x >> y >> z) point = Point{x,y,z}; 
	else Point{}; // Reset point.
	
	return in;
}
```
- Danger: partial extraction
	- Solution: make the operation **transactional** -> must either **completely succeed** or **completely fail** (see above)
- What if extractable but invalid? ie. `std::cin`NOT placed in failure mode.
	- Check manually, then if invalid, setstate to fail
	- `in.setstate(std::ios_base::failbit);`

## Overloading with member functions
- Overloaded operator must be added as member function of left operand
- Left operand becomes the implicit `*this` object
- All other operands become function parameters

```cpp
class Cents {
private:
    int m_cents {};

public:
    Cents(int cents)
        : m_cents { cents } { }

    // Overload Cents + int
    Cents operator+(int value) const;

    int getCents() const { return m_cents; }
};

// note: this function is a member function!
// the cents parameter in the friend version is now the implicit *this parameter
Cents Cents::operator+ (int value) const {
    return Cents { m_cents + value };
}
```
- Comparison with friend function
	- `Cents::operator+ (const Cents& cents, int value)`
		- Call evaluates to `operator+(cents1, 2)`
- Here, 
	- Call evalutes to `cents1.operator+(2)`
		- Remember that the compiler implicitly converts object prefix to leftmost param named `*this`  
			- So it becomes `operator+(&cents1, 2)` which is similar to the friend version

- Restrictions
	- Member functions only
		- =, [], ->
	- Friend functions only
		- <<, or +(int, Cents)
			- The left operand has to be a class we can add members to!
			- Can't use friend function if left operand is not a class (int) or class we can't modify (std::ostream)

> Use friend functions if dealing with binary perators that don't modify the left operand (eg. operator+)
> Use member functions if left operator modified (eg. operator+=), or for unary operators
