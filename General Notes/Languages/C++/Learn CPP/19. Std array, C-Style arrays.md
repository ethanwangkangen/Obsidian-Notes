# Std::array
- Use when we need **constexpr** arrays, since std::vector doesn't allow.
```cpp
std::array<int, 5> a {}; // std::array of 5 ints

std::array<int, 6> fibonnaci = { 0, 1, 1, 2, 3, 5 }; // copy-list initialization using braced list
    std::array<int, 5> prime { 2, 3, 5, 7, 11 };         // list initialization using braced list (preferred)
```

- Should use double braces instead (see below)

- Initialisation
	- **Aggregate initialisation**, since it's an aggregate
	- Should explicitly **value-initialise** `std::array` if not providing initialisation values
		- ie use `{}` -> to ensure that primitives, eg int etc. are zero initialised.
		- `std::array `will **default initialize** members if no initializers are provided. This will leave elements of most types **uninitialized**.
	- Use `static_assert` if possible to ensure a **constexpr** std::array using CTAD has correct number of initialisers
		- `static_assert(arr.size() == 3);`

- Length must be constexpr
	- Can be 0! Special case class that has no data.
	- Cannot use runtime constants or non-constexpr variables
- Can be `const`
	- Elements will be treated as const too, since whole array is const
- Can be `constexpr`, unlike std::vector

> Make std::array **constexpr** if possible, if not consider using std::vector

- CTAD available, C++ 17
	- `constexpr std::array a2 { 9.7, 7.31 }; `

- Omitting array length with `std::to_array`, C++17
```cpp
constexpr auto myArray1 { std::to_array<int, 5>({ 9, 7, 5, 3, 1 }) }; // Specify type and size
constexpr auto myArray2 { std::to_array<int>({ 9, 7, 5, 3, 1 }) };    // Specify type only, deduce size
constexpr auto myArray3 { std::to_array({ 9, 7, 5, 3, 1 }) };         // Deduce type and size
```

## Indexing
```cpp
// Array structure
template<typename T, std::size_t N> // N is a non-type template parameter
struct array;
```
- Index and length are `std::size_t`
- Size with `.size()`
	- Is a **constexpr**. Can be converted to int without narrowing.
- Index with `[]` , `at()`, `get()`
	- [] - no bounds checking
		- `arr[3]`
	- at() - runtime bounds checking
		- `arr.at(3)`
	- get() - compile time bounds checking 
		- `std::get<3>(arr)`

## Passing
- Pass by const reference
	- `void passByRef(const std::array<int, 5>& arr) {}`
- Use function templates to pass array of different element types/lengths
	- Note that the second argument is `std::size_t` not `int`
```cpp
template <typename T, std::size_t N> // note that this template parameter declaration matches the one for std::array
void passByRef(const std::array<T, N>& arr) {...}

// In C++, can use auto non-tyype template params
template <typename T, auto N> // now using auto to deduce type of N
```

## Arrays of structs/classes
- Special case when initialisign with braces
```CPP
struct House {
    int number{};
    int stories{};
    int roomsPerStory{};
};

int main() {
    constexpr std::array<House, 3> houses {{ // note double braces
        { 13, 1, 7 },
        { 14, 2, 5 },
        { 15, 2, 4 }
    }};
    
    houses[0] = { 13, 1, 7 }; // can do this.
    
    // CANNOT!
    constexpr std::array<House, 3> houses { // single braces
        { 13, 1, 7 }, // initializer for C-style array member with implementation_defined_name
        { 14, 2, 5 }, // ?
        { 15, 2, 4 } // ?
    };
    
    
    // OK. brace elision
    constexpr std::array<House, 3> houses { // single braces
        House { 13, 1, 7 }, // element type specified. single brace ok
        House { 14, 2, 5 },
        House { 15, 2, 4 }
    };
    ...
}
```
- Because compiler interprets `{13, 1, 7}` as initialiser for first member of `houses`, which is the C-style array with implementation-defined name
	- So the rest of the members will be zero-initialised

- In fact, for all array list initialisation (aggregate intialisation), we should use double braces. Single brace allowed for primitive types, or when element type is specified, only due to **brace elision**
	- `constexpr std::array<int, 5> arr {{ 1, 2, 3, 4, 5 }};`

## Array of references
- `std::array<int&, 2> refarr {x,y}`
	- NOT allowed! Because references are not objects. The elements of an array must be assignable, while references can't be reseated.
### std::reference_wrapper
- `Operator=` will reseat it (change which object being referenced)
- `std::reference_wrapper<T> `will **implicitly** convert to **T&**.
- `get()` to get a  T&
	- Useful when we want to update the value of the object being referenced.

```cpp
int main() {
    int x { 1 };
    int y { 2 };

    std::array<std::reference_wrapper<int>, 2> arr { x, y};
    arr[1].get() = 5; // .get() since we want to specify that we want to change the value being referenced and not reseat the wrapper.

    std::cout << arr[1] << y << '\n'; // compiler realises it can't print a `std::reference_wrapper<int>` so implicitly converts to int&. no need get()

}
```

### std::ref() and std::cref()
- shortcuts to create `std::reference_wrapper` and `const std::reference_wrapper` wrapped objects
```cpp
int x { 5 };
auto ref { std::ref(x) };   // C++11, deduces to std::reference_wrapper<int>
auto cref { std::cref(x) }; // C++11, deduces to std::reference_wrapper<const int>
```

std::array<std::array<int, 4>, 3> arr {{  // note double braces

# C-style array
```cpp
int testScore[30] {};
const int arr[] { 9, 8, 7, 6, 5 }; // aggregate initialisation
```
- Initialisation
	- Aggregate initialisation: see above. Preferred, since C-Style arrays are aggregates
		- If less elements provided than length, the rest are value initialised.
	- If no intialiser provided, elements default initialised -> most elements left unitialised
		- Thus, initalise c-style arrays with `{}`
- Length
	- Length must be at least 1 (except if dynamically allocated on heap, can be 0)
	- Array length must be **constexpr** (of type std::sze_t)
	- Getting length
		- `#include <iterator>` // for size and ssize()
		- `std::size(arr)` - unsigned integral type
		- `std::ssize(arr)` - C++20, signed integral type
- Indexing
	- Unlike stl containers, index can be value of **any interal tpe**
	- `[]` does not do bounds checking
- Can be `const` and `constexpr`
	- `const` arrays must be initalised and value of elements cannot be changed afterwards
- Size
	- `sizeof(arr)`
	- A C-style array object contains its elements and nothing more, ie. size of an array containing 2 ints each 4 bytes, has total size 8 bytes.

- Assignment not allowed
	- Because assignment requires left operand to be modifiable lvalue.
		- C-style arrays are not considered modifiable lvalues.
```cpp
int arr[] { 1, 2, 3 }; // okay: initialization is fine
arr[0] = 4;            // assignment to individual elements is fine
arr = { 5, 6, 7 };     // compile error: array assignment not valid
```

#### C-style strings
C-style strings
- C-style arrays whose element type is `char` or `const char`
- End with null terminator byte

C-style symbolic constants
```cpp
const char name[] { "Alex" };        // case 1: const C-style string initialized with C-style string literal
const char* const color{ "Orange" }; // case 2: const pointer to C-style string literal
```
- First case: "Alex" in read only memory somewhere, program then allocates memory for C-style array ofr lenght 5 then initialises memory with that string. Thus 2 copies. 
- Second case: implementation defined. Usually, compiler places "Orange" into read onyl memory, then initialiser pointer with address of that string

Outputting
```cpp
char c{'A'};
std::cout << &c;
```
- Garbage printed! 
	- cout **assumes char * passed in is a string, will read until null termination!**
	- If want to output a `char *`, `static_cast<const void*>(ptr)`;



> Avoid C-style string symbolic constants, prefer **constexpr std::string_view**

## Array decay
- In most cases, when C-style array used in **expression**,
	- the array is implicitly conerted into **pointer of element type**, intialised with address of the first element
```cpp
int arr[5]{ 9, 7, 5, 3, 1 }; // array has elements of type int
auto ptr{ arr }; // evaluation causes arr to decay, type deduction should deduce type int*
```

- Cases where no decay
	- Used as argument to sizeof() or typeid()
	- Taking address of array with operator&
	- Passed as a member of class type
	- Passed by reference

- Subscripting a C-style array actually applies `operator[]` to the decayed pointer
```cpp
const int arr[] { 9, 7, 5, 3, 1 };
std::cout << arr[2]; // subscript decayed array to get element 2, prints 5

const int* ptr{ arr };  // arr decays into a pointer
std::cout << ptr[2];    // subscript ptr to get element 2, prints 5
```

- As such, C-style arrays **passed by address**, ie NOT COPIED!! When passing into function
```cpp
void printElementZero(const int arr[]) { // treated the same as const int*
    std::cout << arr[0];
}

// same as the above. BUT use the above as its clearer.
void printElementZero(const int* arr) { // pass by const address
    std::cout << arr[0];
}
```

Problems
- Decayed pointers lose size of array information
- Can't use range based for loops anymore since the size of array is unknokwn

> Avoid C-style pointers if practical
- Prefer `std::string_view` for read-only strings (string literal symbolic constants and string parameters).
- Prefer `std::string` for modifiable strings.
- Prefer `std::array` for non-global constexpr arrays.
- Prefer `std::vector` for non-constexpr arrays.

> Use cases in C++: to store **constexpr global/ constexpr static local** data, or as parameters to functions/classes that want to handle non-constexpr **C-style string** arguments directly (and don't want to convert to std::string_view)

### Pointer arithmetic
- Returns address of the **next/previous** object **based on the pointer type**
	- eg. if pointer to an int, and int is 4 bytes, increment by 4.
- Array subscripting is done via pointer arithmetic
	- `ptr[2]` is equivalent to `*((ptr) + (2))`, which is equivalent to `*(ptr + 2)`. 
	- `ptr + 2 `returns the address of the object that is two objects past ptr, which is the element with index 2. 
	- The object at that address is then returned to the caller.
- Because of this, we can also do
	- `n[ptr]` instead of `ptr[n]`
		- eg. `2[arr]`
	- Since `*((ptr) + (n))` == `*((n) + (ptr))`
- Pointer arithmetic and subscripting are relative addresses
	- eg. if `ptr` is already pointing to element 3, eg. `ptr=&arr[3]` 
		- `ptr[1]` then gives the 4th element.

Use case: iterating through C-style array without explicit indexing
```cpp
#include <iostream>

int main()
{
	constexpr int arr[]{ 9, 7, 5, 3, 1 };

	const int* begin{ arr };                // begin points to start element
	const int* end{ arr + std::size(arr) }; // end points to one-past-the-end element

	for (; begin != end; ++begin) {          // iterate from begin up to (but excluding) end
		std::cout << *begin << ' ';     // dereference our loop variable to get the current element
	}

	return 0;
}
```

### Multidimensional C-style arrays
- Row major order, ie. elements in memory laid out row by row
- When using inner braces, missing initialisers are value initialised
```cpp
int array[3][5] {
  { 1, 2, 3, 4, 5 },     // row 0
  { 6, 7, 8, 9, 10 },    // row 1
  { 11, 12, 13, 14, 15 } // row 2
};

int array[3][5]{
  { 1, 2 },          // row 0 = 1, 2, 0, 0, 0
  { 6, 7, 8 },       // row 1 = 6, 7, 8, 0, 0
  { 11, 12, 13, 14 } // row 2 = 11, 12, 13, 14, 0
};
```


# Iterating, and iterators
## Pointers as iterators
```cpp
std::array<int, 7> arr {1, 2, 3, 4, 5, 6, 7}
int& ptrStart {&arr[0]}; 
// note: can't do &arr, because pointer arithmetic will fail.
int& ptrEnd {ptrStart + std::size(arr)};
// note: int& ptrEnd {&arr[std::size(arr)]} error! dereferencing outside element. *(arr + std::size(arr)) is outside array, dereferencing is undefined, so is taking address there.


for (auto ptr{ptrStart}; ptr != ptrEnd; ptr++) {
	std::cout << *ptr<< ' ';
}
```

## Iterators
- `.begin()`, `.end()`
- Essentially like pointers
	- For std::vector and arrays, are literally a pointer
	- For other containers, may be class object that wraps a pointer toa  node (eg. linked list), with overloaded operators (`*, ++, --`) to make it behave like pointer
```cpp
std::vector<int> v{10, 20, 30};

// Use auto (recommended)
auto it = v.begin(); // iterator to first element

std::cout << *it << "\n"; // prints 10

++it; // move to next element
std::cout << *it << "\n"; // prints 20

// Explicit type
std::vector<int>::iterator it2 = v.begin();
```

Operators
- Dereference `*` 
- Member access `->`
- Incremement `++`
- Some iterators (like vector/array) support random access
	- `+n`, `[n]`

Comparison
- Use `!=` instead of `<`, since some iterator types may not support the latter

Invalidation
- Dangling iterators, if elements being iterated over change address or are destroyed
- eg.
	- If we do `push_back()` while iterating through a vector. invalidates the iterators of vector, and range-based for-loops use iterators behind the scenes, so undefined behaviour
	- `vector.erase(it)` -> erases element currently being iterated over, but sets 'it' to the next element. so can do
		- `it = v.erase(it)` //ok.
