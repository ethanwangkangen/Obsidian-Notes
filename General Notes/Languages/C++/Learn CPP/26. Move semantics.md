# Auto pointer
- Wrapper for a type T, that holds the pointer to the object.
- When auto pointer is destroyed, it **deletes** the pointer too
- When it is copied with copy constructor,
	- **Move** the pointer from source to local object
	- ie. Take ownership of the pointer.
- When another object is **assigned** this object,
	- **Move** our pointer to that object.
	- Set this pointer to null.
```cpp
#include <iostream>

template <typename T>
class Auto_ptr2{
	T* m_ptr {};
public:
	Auto_ptr2(T* ptr=nullptr)
		:m_ptr(ptr)
	{
	}

	~Auto_ptr2() {
		delete m_ptr;
	}

	// A copy constructor that implements move semantics
	Auto_ptr2(Auto_ptr2& a) // note: not const
	{
		// We don't need to delete m_ptr here.  
		// This constructor is only called when we're creating a new object
		// so m_ptr can't be set before this.
		m_ptr = a.m_ptr; // transfer pointer from a to this obj
		a.m_ptr = nullptr; // make sure a no longer owns the pointer
	}

	// An assignment operator that implements move semantics
	Auto_ptr2& operator=(Auto_ptr2& a) { // note: not const
		if (&a == this)
			return *this;

		delete m_ptr; // deallocate destination's pointer
		m_ptr = a.m_ptr; // transfer our pointer to theirs
		a.m_ptr = nullptr; // make sure the source no longer owns the pointer
		return *this;
	}

	T& operator*() const { return *m_ptr; }
	T* operator->() const { return m_ptr; }
	bool isNull() const { return m_ptr == nullptr; }
};
```
- Why deprecated/bad idea?
	- Passing auto ptr by value into a function will implement move semantive, even through the copy constructor.
		- Then resource gets moved into the function paramter and destoyed at the end of the function
			- We may not have wanted this! End up dereferencing null pointer when we access the auto ptr argument from caller
	- Deletes its contents using **non-array delete**. Won't work correctly with dynamically allocated arrays.

## R-value references
- R-value reference

| R-value reference       | Can be initialized with | Can modify |
| ----------------------- | ----------------------- | ---------- |
| Modifiable l-values     | No                      | No         |
| Non-modifiable l-values | No                      | No         |
| R-values                | Yes                     | **Yes**    |

- Rvalue reference to const

|R-value reference to const|Can be initialized with|Can modify|
|---|---|---|
|Modifiable l-values|No|No|
|Non-modifiable l-values|No|No|
|R-values|Yes|No|

eg.
```cpp
auto&& rref{ Fraction{ 3, 5 } }; // r-value reference to temporary Fraction
```
- Temporary object created from the literal, then the reference can modify that temporary object (same like with lvalue reference to const)

- Use case
	- Function parameters
		- Useful for function overloads, if want differnet behaviour for l-value and r-value arguments

> IMPORTANT! **rvalue reference variables are lvalues**

```cpp
int&& ref{ 5 };
fun(ref); 
// fun(const int&) or fun(int&&) ?
```
- The latter is chosen!
	- ref is type `int&&`, but when **used in an expression**, is an **lvalue**
		- Because it is named!
	- It can't even choose the latter, since rvalue references can't bind to lvalues

# Move constructors and assignment
- Recall auto pointer example
- Move constructor and move assignment
	- Instead of creating new objects dynamically, just move ownership of the same resource from one object to another

- Copy constructor vs Move constructor
	- Copy constructor
		- In the former, `m_ptr` points to a dynamically allocated T object.
		- Then, this object is **copy assigned from** the object that a's m_ptr points to.
	- Move constructor
		- We just take over that pointer.
```cpp
// Copy constructor
	// Do deep copy of a.m_ptr to m_ptr
	Auto_ptr4(const Auto_ptr4& a) {
		m_ptr = new T;
		*m_ptr = *a.m_ptr;
	}

// Move constructor
	// Transfer ownership of a.m_ptr to m_ptr
	Auto_ptr4(Auto_ptr4&& a) noexcept
		: m_ptr { a.m_ptr } {
		a.m_ptr = nullptr; // we'll talk more about this line below
	}
```

- Copy assignment vs Move assignment
	- Copy assignment
		- Similarly, `mp_ptr` points to new dynamically allocated object.
		- Then that object is **copy assigned from**  the object that a's m_ptr points to
	- Move assignment
		- Take over the pointer.
```cpp
// Copy assignment
	// Do deep copy of a.m_ptr to m_ptr
	Auto_ptr4& operator=(const Auto_ptr4& a) {
		// Self-assignment detection
		if (&a == this)
			return *this;

		// Release any resource we're holding
		delete m_ptr;

		// Copy the resource
		m_ptr = new T;
		*m_ptr = *a.m_ptr;

		return *this;
	}
	
// Move assignment
	// Transfer ownership of a.m_ptr to m_ptr
	Auto_ptr4& operator=(Auto_ptr4&& a) noexcept {
		// Self-assignment detection
		if (&a == this)
			return *this;

		// Release any resource we're holding
		delete m_ptr;

		// Transfer ownership of a.m_ptr to m_ptr
		m_ptr = a.m_ptr;
		a.m_ptr = nullptr; // we'll talk more about this line below

		return *this;
	}
```
- Make sure to set a's `m_ptr` to `nullptr`!
	- Because when a goes out of scope, it's destructor will be called which will try to free the `m_ptr`, so it better not be pointing to something that it doesnt own.

## Disabling copying
- Delete the default copy constructor
	- Make sure to include `const`! Because that's the function signature of the standard copy constructor
```cpp
Auto_ptr5(const Auto_ptr5& a) = delete;
```

## Disabling moving
- If delete copy constructor, compiler will **not** generate an implicit move constructor
```cpp
Name& operator=(Name&& name) = delete;
```

# Rule of 5
- If copy constructor/copy assignment/move constructor/move assignment/destructor are defined or deleted, then all of those should be defined or deleted

- Why can't delete move constructor and move assignment, and leave copy constructor and assignment? To make object copyable only?
	- **Return from value favour calling the deleted move constructor, so error**
	- Will no longer be able to return this object type as return value from a function.

# Copy vs move function signature summary
```cpp
copyFunction(const T& value); // copy. accepts l-value, both const and non const, and r-values also (temp created)
moveFunction(T&& value); // accepts r-values only
```
- Note that while `const T&` can bind to r-value, the latter is preferred during resolution, since C++ overload resolution uses reference binding rules
	- Exact match beats binding to const
	- T&& matches r value exactly
	- const T& requires binding a const reerence to an rvalue, less exact
# std::move
- Moves l-value object into an r-value reference, so that move assignment/constructors can use it
ie
```cpp
std::string str = "hello";

std::move(str);
// is basically
static_cast<std::string&&>(str);

// allowing things like
v.push_back(std::move(str));
// overloaded function that takes in r-value
```

- eg. 
```cpp
template <typename T>
void mySwapCopy(T& a, T& b) {
	T tmp { a }; // invokes copy constructor
	a = b; // invokes copy assignment
	b = tmp; // invokes copy assignment
}

template <typename T>
void mySwapMove(T& a, T& b) {
	T tmp { std::move(a) }; // invokes move constructor
	a = std::move(b); // invokes move assignment
	b = std::move(tmp); // invokes move assignment
}
```

- **Moved-from objects** will be in a valid but possibly **indeterminate state**
	- It is safe to call any function that **does not depend on the current value of the object**
		- Can **set, reset value, operator=, clear(), reset() etc.**

# std::unique_ptr
```cpp
#include <memory> // for std::unique_ptr

std::unique_ptr<Resource> res1{ new Resource() };
std::unique_ptr<Resource> res2{}; // Start as nullptr
res2 = std::move(res1);
```
- Like auto_ptr, 
	- Owns a pointer that points to the resource

- Accessing
```cpp
std::cout << *res; // dereferences the pointer that unique_ptr stores
res.get(); // returns the pointer itself
```

- Implicit **cast** to bool to check if it contains a resource
```cpp
if (res) {...}
```

- Creating a unique_ptr
	- `std::make_unique<T>`
	- Better than creating a `unique_ptr` yourself
```cpp
auto f1{ std::make_unique<Fraction>(3, 5) };
std::cout << *f1 << '\n';

// dynamically allocated array of Fractions of length 4
auto f2{ std::make_unique<Fraction[]>(4) };
std::cout << f2[0] << '\n';
```

- Returning unique_ptr
	- If assigned, move semantics applied
	- Don't return `std::unique_ptr` by pointer (ever) or reference (unless you have specific compelling reason to)
```cpp
std::unique_ptr<Resource> createResource() {
     return std::make_unique<Resource>();
}
```


- Passing std::unique_ptr to a function
	- If want the function to **take ownership** of the **contents of the pointer**, pass it **by value**
	- Must use **move semantics** since copy semantics disabled
```cpp
// This function takes ownership of the Resource, which isn't what we want
void takeOwnership(std::unique_ptr<Resource> res) {
     if (res)
          std::cout << *res << '\n';
} // the Resource is destroyed here

//takeOwnership(ptr); // Not allowed since this calls copy constructor which is disabled for unique_ptr
takeOwnership(std::move(ptr)); // ok: use move semantics
```

- If we **DON'T want function to take ownership** (which is usually),
	- Just pass the resource itself (by pointer or reference, depending on whether null is valid argument)
```cpp
// The function only uses the resource, so we'll accept a pointer to the resource, not a reference to the whole std::unique_ptr<Resource>
void useResource(const Resource* res) {
	if (res)
		std::cout << *res << '\n';
	else
		std::cout << "No resource\n";
}
useResource(ptr.get()); // note: get() used here to get a pointer to the Resource
```

## What NOT to do
- DON'T let multiple objects manage the same resouce
```cpp
Resource* res{ new Resource() };
std::unique_ptr<Resource> res1{ res };
std::unique_ptr<Resource> res2{ res };
```

- DON'T manually delete the resource within the std::unique_ptr
```cpp
Resource* res{ new Resource() };
std::unique_ptr<Resource> res1{ res };
delete res;
```

# std::share_ptr
- When you need multiple smart pointers co-owning a resource
```cpp
Resource* res { new Resource };
std::shared_ptr<Resource> ptr1{ res };
std::shared_ptr<Resource> ptr2 { ptr1 }; // points to same resource!
// NOT std::shared_ptr<Resource> ptr2 { res };
```

- Stores 2 pointers internally
	- One pointer points at resource being managed
	- Other points at 'control block' -> dynamically allocated object that tracks things like std::shared_ptr 
	- Each std::shared_ptr will independently allocate its own control block
	- When no more poiners to it, then can safely deallocate the main pointer

- BUT!!!! Must instantiate ptr2 with ptr1
	- If not, ptr2 is not aware of ptr1, and the double free will cause program to crash

- Making
	- Do it this way. Easier and more performant
```cpp
auto ptr1 { std::make_shared<Resource>() };
```

- Can be created from unique pointers

- Dereferencing share dpointers
- Eg.
```cpp

auto lucy { std::make_shared<Person>("Lucy") }; // create a Person named "Lucy"

…

p1->m_partner = p2;
```

- p1-> // this called the operator->() overloaded function on shared pointer, which returns the internal pointer to the Lucy object.
	- Then, -> happens on that pointer, dereferencing it to get the actual Lucy object, and then accessing the m_partner member.
	- Ie (Person*)->m_partner


  

## Circular dependency issues
- eg.2 Objects that hold shared pointers to each other
- Cannot deallocate, even when objects go out of scope. When A goes out of scope, it checks if there are shared pointers that co-own A, in this case yes since B, so does not deallocate A. Then likewise to B.    


- Thus this happens when each object references the next, and the last object references back to the first, causing a referential loop.    


# Weak_ptr
- Designed to solve the above issue.
- Can observe and access the same object as a std::shared_ptr, but is not considered an owner
- When another std::shared pointer goes out of the scope, it will not consider this weak_ptr as co-owning the object.
- Can only observe an object that is already owned by a shared_ptr.
- Cannot dereference directly, must convert to a std::shared_ptr first.
	- Do this with .lock()
	
- Will not keep an owned resource alive, so possible for dangling pointer here!

- Test with the expired() member function

- Returns true if pointing to invalid object, else false

```cpp
auto sharedPtr1 {std::make_shared<Resource>()}; // shared_ptr to resource

std::weak_ptr<Resource> wkPtr1 = sharedPtr1; // remember what this assignment does for shared_ptr! It copies over the control block, so same internal pointer to the same object, and increments reference count. For weak_ptr, it just references the control block and does not increment reference count
```
