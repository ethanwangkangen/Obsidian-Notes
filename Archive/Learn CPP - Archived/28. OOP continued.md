# Pointers and references to base class of derived objects
- Unable to see anything in the derived class!
- Use cases
	- Containers for different types that inherit from the same base class
		- Store pointers to their base class
```cpp
const auto animals{ std::to_array<const Animal*>({&fred, &garbo, &misty, &pooky, &truffle, &zeke }) };
```

- Functions for different types that inherit from the same base class (instead of a new overload for each)
```cpp
void report(const Animal& rAnimal) {
    std::cout << rAnimal.getName() << " says " << rAnimal.speak() << '\n';
}
```
- Side: can do this too instead
```cpp
template <typename T>
void report(const T& rAnimal) {
    std::cout << rAnimal.getName() << " says " << rAnimal.speak() << '\n';
}
```

## Downsides
- Pointer/reference to base class calls **base** version of function instead of derived version, since doesn't have access to the derived version

# Virtual functions - runtime polymorphism
- Resolves to the **most-derived version** of the function for the **actual type** of the object it belongs to
	- (Start from the exact one, then go up. Similar to Java runtime overloading)
		- But this is not what actually happens. Will explain later one.
	- Derived function considered match if has same function signature (name, param types, const) **AND RETURN TYPE** as base version
	- These functions are called overrides
```cpp
class Base{
public:
    virtual std::string_view getName() const { return "Base"; } // note addition of virtual keyword
};
```
- Virtual function resolution only works when member function is called **through pointer or reference** to a class object
- If a function is marked as virtual, **all matching overrides** in derived class are **implicitly virtual**, even if not explicitly marked

- **Do not call** virtual functions from **constructors/destructors**
	- Constructor
		- If call virtual function from Base constructor, Derived portion not created yet, so will just call the Base version
		- If call virtual function from Base destructor, Derived portion already destroyed, so will just call Base version

## Override specifier
- This doesn't change any runtime behaviour at all! Is just used as a **safety check** to ensure that you are actually overriding functions that you think you are
- Enforces this rule
	- This function must **match a virtual function in a base class exactly**
		- Same name, param types, const/ref qualifiers, correct inheritance chain

## Final specifier
- Restrict user from further overriding a function later down the line
	- This function cannot be overridden in a derived class
	- Can be used with 'override'
- Can be used with classes too

```cpp
// Cannot inherit from B 
class B final : public A {
public:
	std::string_view getName() const override { return "B"; }
};

// Cannot override C::getName()
class C : public A {
public:
	std::string_view getName() const override final { return "C"; } 
};
```
## Virtual destructor
- **Always make destructor virtual when class is inherited!!!!**
	- To ensure that the derived destructor is called, which helps to clean up resources belonging to the derived part

- If class isn't explicitly designed to be a base class, better to have **no** virtual member and no virtual destructor. 
	- If do not intend class to be inherited from, mark it as final
- If class is desigend to be base class and/or has any virtual functions, then it should always have a virtual destructor, and make it public
## Virtual assignment
- Rarely a good idea. KIV

## Ignoring virtualisation
- Say we have a Base Pointer to a Derived object, but want to call Base::getName() instead of Derived::getName() which has been labelled virtual
	- Use scope resolution operator
```cpp
Derived derived {};
const Base& base { derived };

// Calls Base::getName() instead of the virtualized Derived::getName()
std::cout << base.Base::getName() << '\n';
```

# Binding and dispatching
- Binding: associating names with properties
- Function binding: process that determines what function definition is associated with a fcuntion call

## Early binding
- Direct call made to non-member function / non-virtual member function
	- Resolved at compile time
```c
// assembly code
mov     edi, 5           ; copy argument 5 into edi register in preparation for function call
call    printValue(int)  ; directly call printValue(int)
```
- Overloaded functions can also be resolved via early binding

## Later binding
- Function call can't be resolved until runtime, ie. not during compile time.
- Examples
	- Function pointer
```cpp
auto fcn { printValue }; // create a function pointer and make it point to function printValue
fcn(5);        

// corresponding assembly code
lea     rax, [rip + printValue(int)] ; determine address of printValue and place into rax register
mov     qword ptr [rbp - 8], rax     ; move value in rax register into memory associated with variable fcn

mov     edi, 5                       ; copy argument 5 into edi register in preparation for function call
call    qword ptr [rbp - 8]          ; invoke the function at the address held by variable fcn         
```

- Vtables
	- Dynamic dispatch
	- Every class that uses virtual functions (or is derived from class that uses virtual functions) has a corresponding vtable
		- **NOT each object!** One vtable per class. Then the object has a pointer that points to that table.
	- One entry for each virtual function that can be called by objects of the class
		- Contains function pointer that points to the **most-derived function** accessible by the class
	- Compiler adds a hidden pointer that is member of bass class, which is `*__vptr`, ie pointer to the virtual table
		- Inherited by classes that inherit it
		- But the value it holds is different (since it points to a different Vtable)

Calling virtual function - SLOW
- Need to use `*__vptr` to get the approriate virtual table
- Then need to index the virtual table to find the correct function to call
- Then need to go to that address tocall that function

Note: **using the scope resolution operator (::) BYPASSES the vtable!**

## Pure virtual functions
```cpp
virtual int getValue() const = 0; // a pure virtual function
```
- Any class with at least one virtual functions becomes **abstract base class**
	- Cannot be instantiated
- Any class with pure virtual function **should also have virtual destructor**

- Abstract classes still have virtual tables
	- Generally contain null pointer, or point to generic function that prints an error
### Interface class
- Class that has **no member variables** and **all functions are pure virtual**
- Similar to interface classes in Java

## Virtual base class
- Diamond inheritance problem
	- 2 instances of the first base class constructed!
![[Pasted image 20260413152816.png]]

- Virtual base class
	- Only one base object
	- Shared between all objects in the inheritance tree and is only constructed once
	
```cpp
class PoweredDevice {
};

class Scanner: virtual public PoweredDevice {
};

class Printer: virtual public PoweredDevice {
};

class Copier: public Scanner, public Printer {
};
```
- Scanner, Printer, **AND COPIER** all instantiate a PoweredDevice in their constructor member initialiser list
	- BUT **only copier** actually constructs the PoweredDevice, (when copier is constructed)
	- If we instantiate Printer directly, then of course Printer constructs the PoweredDevice
	- Also this is the one time when Copier is allowed to call a non-immediate-parent constructor directly

- For constructor of most derived class, virtual base classes always created before non-virtual base classes
	- Ensures all bases get created before derived classes

- If class inherits one or more classes that have virtual parents, the **most derived class** is the one responsibel for construcing the virtual base class (even in singae inheritance case)

- All classes inhering virtual base class will have a virtual table, even if they normally would not have one otherwise

# Object slicing
```cpp
Derived derived{ 5 };
Base base{ derived }; // what happens here?
```
- Derived object assigned to Base oject
	- **Only the Base portion** is copied

- Be careful with function parameters!
	- Eg. 
```cpp
void func (const Base base);
Derived d{5};
func(d); // accidental slice!

// To solve this, make func take in a reference instead
void func (const Base& base); // Base reference view into a Derived object.
// Virtual functions will resolve correctly too.
```

### Slicing vector
```cpp
std::vector<Base> v{};
v.push_back(Base{ 5 });    // add a Base object to our vector
v.push_back(Derived{ 6 }); // add a Derived object to our vector
```
Solution
- vector of `Base*`
- vector of reference wrappers

### Franken object problem
```cpp
Derived d1{ 5 };
Derived d2{ 6 };
Base& b{ d2 };

b = d1; // this line is problematic
```
- b points at d2
- Assign d1 to b 
	- b is a Base, and the assignment operator that C++ provides for classes is not virtual
	- So assignment that copies a **Base** is invoked
	- **Only the Base portion of d1 is copied into d2**
- Now d2 has the Base portion of d1 and the Derived portion of d2.

> If Base class is not designed to be instantiated by itself (eg. just an interface class), slicing can be avoided by making the Base class non-copyable

## Dynamic casting
- Have pointer to **base class**, but need information that exists only in derived class
	- Downcast: pointer/reference from **base type -> derived type**
```cpp
Base* getDerived() {
	return new Derived{1, "Apple"};
}

Base* b{ getDerived() };
Derived* d{ dynamic_cast<Derived*>(b) }; // use dynamic cast to convert Base pointer into Derived pointer

// Can also do this with references
Derived apple{1, "Apple"};  
Base& b{ apple }; 
Derived& d{ dynamic_cast<Derived&>(b) }; // dynamic cast using a reference instead of a pointer
```
- Dynamic_cast checks that the object is actually what it's casting to!
	- If cast fails, returns **null pointer**
	- Checks that b is polymorphic (**must have at least one virtual function**)
	- Follow vptr to get the object's actual runtime type, then compare
		- Because vtable contains type information!
- Does not work in some scenarios
	- With protected/private inheritance
	- Classes that do not declare or inherit any virtual functions (no vtable)
	- Certain cases involving virtual base classes
```cpp
if (d) ... // make sure dynamic cast succeeds!
```

## Static casting
- Can downcast statically too, but no check!
```cpp
Derived* d{ static_cast<Derived*>(b) }; // must be sure that b is pointing to a Derived object
```

> In general, use static_cast unless downcasting

> In general, avoid dynamic casting and use virtual functions. Unless the derived function doesn't make sense to put in the base class.


## Visitor 
- Instead of type based branching
```cpp
if (auto c = dynamic_cast<Circle*>(shape)) { ... }
else if (auto s = dynamic_cast<Square*>(shape)) { ... }
```
- We want a way to have dynamic resolution
	- But virtual functions in Shape will clutter Shape, especially if there are a lot of operations
	- Solutions: use a Visitor

```cpp
struct Circle;  
struct Square;  
  
struct ShapeVisitor {  
	virtual void visit(Circle&) = 0;  
	virtual void visit(Square&) = 0;  
	virtual ~ShapeVisitor() = default;  
};

struct Shape {  
	virtual void accept(ShapeVisitor&) = 0;  
	virtual ~Shape() = default;  
};

struct Circle : Shape {
    void accept(ShapeVisitor& v) override {
        v.visit(*this);
    }
};

struct Square : Shape {
    void accept(ShapeVisitor& v) override {
        v.visit(*this);
    }
};

struct RenderVisitor : ShapeVisitor {
    void visit(Circle& c) override {
        // render circle
    }

    void visit(Square& s) override {
        // render square
    }
};

int main() {
	Shape* s = getShape(); // might point to a Circle or a Square object
	RenderVisitor renderer;

	s->accept(renderer);
}
```
- `Shape* s` either points to Circle or Square object
- `s->accept(renderer)` 
	- Virtual dispatch happens here.
	- vtable chooses `Circle::accept`, for eg. if s points to Circle
- `Circle::accept()` calls `v.visit(*this)`, which chooses the overload that takes in a Circle reference.

- **Double dispatch** paradigm

|Mechanism|Type resolved|When|
|---|---|---|
|virtual call (`accept`)|Shape type|runtime|
|overload (`visit`)|Visitor function|compile time|
