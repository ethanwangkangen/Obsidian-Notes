//Synax
```cpp
//[capture clause] (param) -> return type {statements}

eg.
auto divide = [](int a, int b) -> double {return (double) a/b;}; 
// can also initialise divide with {}
```
- Can store lambdas in functoin pointer **if it has no captures**
- However, lambdas don't actually have a type
	- Compiler generates a type **just for that lambda** (each lambda has unique type)
	- Lambdas are not actually functions. Are **functors**
		- Like structs with operator()() so can call with ()

## Passing lambdas to functions
```cpp
// 1. std::function
void repeat1(int repetitions, const std::function<void(int)>& fn))

// 2. function template with type template param
template <typename T>
void repeat2(int repetitions, const T& fn)

// 3. abbreviate function template syntax (C++ 20)
void repeat3(int repetitions, const auto& fn)

// 4. function pointer (only for lambda with no captures
```cpp
void repeat4(int repetitions, void (*fn)(int))
```
- 1. std::function
	- Requires lambda to be **implicitly converted** whenever function is called, thus overhead.
- 2,3. Function template
	- When function called, function insantiated, where T matches the actual type of the lambda.
		- T = `<lambda's unique type>`
	- Why **const reference**?
		- Because lambda is a **temporary object (rvalue)**, so need const T& to bind to temporary (life time extended)
		- **Special case: mutable lambdas (KIV)**
			- Use T&& instead.
				- Note: T&& in templates are not just rvalue references, they are **forwarding references**. Can bind to **both rvalue and lvalue**, not just the former like T&& usually is limited to.
				- Reference collapsing + type deduction
- 4. Function pointer
	- Lambda with no captures implicitly converts to function pointer


## Constexpr lambdas
- Lambdas **implicitly constexpr** if **result** satisfies requirements of constexpr
	- Lambdas must have no captures, or all captures constexpr
	- Functions called by lambda must be constexpr

## Generic lambdas
- Lambdas with **auto** parameters
	- In C++20, can use auto params in regular functions too.
- eg.
```cpp
[](const auto& a, const auto& b) {return a[0] == b[0]};
```

- Unique lambda generated for each different type that auto resolves to
	- If contain static variables, each unique lambda will have its own independent static local variable

## Return types
- If explicitly stated, compiler will do implicit conversions
- If not, all return types must match exactly. May have to do explicit conversions to make them all match

## Captures
- Lambdas can only access certain kinds of objects that have been defined outside lambda, including
	- Objects with static storage duration (global variables, static locals)
	- Objects that are constexpr (explicitly or implicitly)
- If don't fulfil the above, need use **capture clause**
	- Used to give lambda access to variables in surrounding scope it would not normally have access to

```cpp
int main() {
  std::array<std::string_view, 4> arr{ "apple", "banana", "walnut"};

  std::string search{}; // Not constexpr!
  std::cin >> search;

  // Capture @search                                vvvvvv
  auto found{ std::find_if(arr.begin(), arr.end(), [search](std::string_view str) {
    return str.find(search) != std::string_view::npos;
  }) };
```

- Captures create a **clone** of that variable inside the lambda
	- Don't actually have the same type as the origiaanal variable
	- Remember that lambdas are functors. Captured variables become data member of object
- Captures treated as **const** by default
	- Can't modify them within the function
- Capture more then one variable, separate with **,**

### Mutable captures
- To allow modify of captured variables, need mark as `mutable`
	- But remember that it modifies its **own cloned copy**, not the outside copy.
	- Also take note that **captured variables value persist across multiple calls to lambda!**
		- Because captured variables are members of the lambda object
```cpp
int ammo{ 10 };

auto shoot{
	[ammo]() mutable { // now mutable
    // We're allowed to modify ammo now
    --ammo;
}
```

### Capture by reference
- Modifies the outer variable
```cpp
int ammo{10};
auto shoot {
	[&ammo]() {--ammo;}
};
```

### Default captures
- Captures **ALL** variables mentioned in the lambda. 
	- If not mentioned, not captured
- `[=]` Capture ALL by **value**
- `[&]` Capture ALL by **reference**
- Mixing
```cpp
// Capture enemies by reference and everything else by value.
[=, &enemies](){};

// Capture armor by value and everything else by reference.
[&, armor](){};

// Illegal, we already said we want to capture everything by reference.
[&, &armor](){};
```

### Defining new variables in lambda capture
```cpp
[userArea{width * height}] (int knownArea) {};
```
> Only initialise variables in capture if value is short and type is obvious, else define outside lambda and capture it.

### Dangling captures
- Variables **captured** at the point where **lambda is defined**
	- If variable captured by reference dies before lambda, lambda left holding dangling reference

```cpp
// returns a lambda
auto makeWalrus(const std::string& name) {
  // Capture name by reference and return the lambda.
  return [&]() {
    std::cout << name;
  };
}

int main(){
  // Create a new walrus whose name is Roofus.
  // sayName is the lambda returned by makeWalrus.
  auto sayName{ makeWalrus("Roofus") };

  // Call the lambda function that makeWalrus returned.
  sayName();
}
```
- The lambda in makeWalrus() captures the temporary string by reference.
	- Temporary string dies after the `auto sayName{ makeWalrus("Roofus") };` expression
	- Then `sayName();` -> reference is dangling.
- Even if we change makeWalrus to pass by value, this still fails
	- `name` dies at the end of `makeWalrus()`
	- lambda is left holding a dangling reference.
- Here, we should capture `name` by value instead.

### Unintended copies of MUTABLE lambdas
```cpp
// Eg.1
auto count{ [i]() mutable {
	std::cout << ++i << '\n';
} };

count(); // >> 1

auto otherCount{ count }; // create a copy of count

count(); // >> 2
otherCout(); // >> 2
```
- When we create copy of count, we create a copy of count **in its current state**
	- Captured `i` =1 for the otherCount.

```cpp
// Eg.2
void myInvoke(const std::function<void()>& fn) {
    fn();
}

int i{0};
auto count {[i]()mutable {std::cout << ++i<<"\n";}};
myInvoke(count); // >> 1
myInvoke(count); // >> 1
myInvoke(count); // >> 1
```

- lambda type doesnt match reference paramter type std::function
	- Lambda converted to temporary std::function so reference parameter can bind to it
	- Thus a **copy** of the lambda is being made.

Solutions
- Put lambda into std::function directly
```cpp
std::function count{ [i]() mutable { // lambda object stored in a std::function
      std::cout << ++i << '\n';
} };
```

- Wrap count in a **std::reference_wrapper**
	- std::ref(count) ensures count is treated like a reference. So anything that tried to copy count will actually copy the **reference**
```cpp
myInvoke(std::ref(count));
myInvoke(std::ref(count));
myInvoke(std::ref(count));
```

> Remember that STL functions may copy function objects (reminder: lambdas are function objects)
> If want to provide lambdas with mutable captured variables, pass them by reference using std::ref

> Try to avoid mutable lambdas.
